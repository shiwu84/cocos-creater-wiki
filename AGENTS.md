# AGENTS.md — LLM Wiki Schema

本文件是 LLM Wiki 的 Schema 配置。`llm-wiki.md` 是概念文档（绝不修改），本文件是具体约定，由 LLM 和用户共同维护。

## 核心理念

本仓库**不是**文档镜像、教程集合、RAG 数据库、API 参考手册。

它是**长期演化的个人知识系统**——通过增量 Ingest 和 Query 归档，持续构建一张跨领域的概念网络。Wiki 的价值在于**连接的密度和质量**，不在于页面数量。同一个概念只在一个地方深入解释；其他页面通过 wikilink 引用它。

### 运行原则

- **编译优先于检索**：raw/ 是事实来源，wiki/ 是已编译的理解层。回答问题时优先使用 wiki/ 中的综合页面，再回到 raw/ 验证细节。
- **增量维护优先于批量重建**：除非用户明确要求重构，否则每次只更新受新知识影响的页面、索引和日志。
- **归档高价值对话**：Query 产生的跨领域综合、对比、排障路径、概念冲突或新连接，不应只留在聊天记录中；应询问用户是否归档到 wiki/。
- **连接质量优先于页面数量**：创建新页前先判断是否能深化已有页；如果新页不能连接到至少 3 个相关页面，优先合并到现有页面或标记为待扩展。
- **人工策源，LLM 维护**：用户决定源材料和研究方向；LLM 负责结构化、交叉引用、一致性检查、索引和日志。

## 架构

```
llm-wiki.md          # 概念参考（只读，绝不修改）
AGENTS.md            # Schema 约定（本文件）
raw/                 # 原始源文档 — 只读，绝不修改
wiki/                # LLM 生成的知识页面 — LLM 全权拥有
output/              # 生成物（幻灯片、图表、canvas 等）
index.md             # Wiki 内容目录（按学习层组织，每次 Ingest 后更新）
log.md               # 操作日志（追加写入）
```

- **禁止**修改 `raw/` 下任何文件。只读取、引用、摘录。
- **禁止**主动修改 `llm-wiki.md`，除非用户明确要求更新 Schema 设计。
- **使用中文回复用户**。
- 对笔记（.md 文件）的读写操作，优先使用 `obsidian-markdown` skill；批量搜索使用 `obsidian-cli`；canvas 文件使用 `json-canvas`；.base 文件使用 `obsidian-bases`。
- 涉及多个文件的操作时，优先使用多智能体并行处理。
- 本仓库使用 **Jujutsu VCS (jj)** 作为版本控制工具，兼容 Git 后端。`jj` 自动快照，无需 `git add`。

### 链 1：游戏开发链

```
Linux → JS → TS → Cocos Creator → Game Dev
```

- **Linux 页面**是运行环境层——解释所有软件运行的底层基础
- **JS 页面**是语言前提层——解释语言核心概念（原型链、闭包、事件循环），不假设上层知识
- **TS 页面**是类型能力层——解释类型系统如何增强 JS，链接回 JS 前提概念
- **Cocos 页面**是引擎应用层——解释引擎如何利用 TS/JS，标注所需前提
- **Game Dev 页面**是实践层——解释实际游戏开发模式和经验，链接回引擎概念

### 链 2：系统编程链

```
Linux → 系统编程概念 → Rust
```

- **Linux 页面**提供 OS 底层理解（进程模型、虚拟内存、文件描述符、系统调用）
- **系统编程桥梁概念**存在于 Linux 和 Rust 页面之间，通过双向链接贯通：如「内存管理」在 Linux 侧讲虚拟内存与页表，在 Rust 侧讲所有权与借用检查器
- **Rust 页面**解释语言如何安全地封装系统编程能力（所有权 = 无 GC 的内存安全，Send/Sync = 编译期并发安全）

### 链接规则

每层页面的「相关页面」应遵循：

| 链接方向 | 含义 | 示例 |
|---------|------|------|
| **向下链接** | 本页面依赖的前提概念 | [[JS 事件循环]] ← [[linux/进程调度]] |
| **向上链接** | 本页面开启的后续概念 | [[JS 事件循环]] → [[Cocos/调度系统]] |
| **跨链链接** | 不同链上存在概念共鸣 | [[Rust Send/Sync]] ↔ [[JS 事件循环]]（都是并发模型） |
| **横向链接** | 引用软件工程通用概念 | [[Cocos/脚本系统]] → [[sw-eng/设计模式：观察者]] |

> 跨链链接只有概念确实存在深层关联时才添加——不是为了链接而链接。

## 页面质量原则

页面的目标是**深度理解**而非**表面摘要**：

1. **自洽**：`> [!abstract]` 部分让读者在不阅读全文的情况下获得完整的核心理解
2. **概念先行**：从"它解决什么根本问题"开始，而非从 API 表面开始
3. **层层深入**：先建立简单心智模型，再添加细节和边界情况
4. **高连接度**：每个页面都有 wikilink 指向前提概念和它开启的概念；跨链链接在概念共鸣处添加
5. **有据可查**：每条断言都能追溯到 `raw/` 源文档或已记录的项目实践
6. **非镜像**：永远不要只是复述 raw/ 文档。要编译、连接、综合。如果页面读起来像文档译稿就需要重写

避免：
- 只有 API 列表没有概念解释的页面
- 所有内容都来自单个源文档的页面（表示做了摘要而非综合）
- wikilink 为零或仅有一个的页面（概念孤岛）
- 超过 30 天未更新且带 `#status/draft` 标签的页面
- **为每篇源文档创建独立页面**（一个源文档 ≠ 一个 Wiki 页面）

## 页面模板（最低要求）

```markdown
---
title: 页面标题
date: YYYY-MM-DD
tags:
  - layer/xxx
  - type/xxx
  - status/xxx
aliases: []
---

# 页面标题

> [!abstract] 摘要
> 一段完整的核心理解——不是目录索引，而是"这一页讲什么、为什么重要"。

## 正文（自由组织）

根据内容需要组织章节，不强制固定标题。但必须包含：
- 概念的**根本问题**（它为什么存在？）
- 与其他系统的**实际关联**（不只是列链接，要解释如何关联）
- 重要的**注意事项和陷阱**

## 相关页面

- [[前提概念1]] — 本页面依赖的前置知识
- [[前提概念2]]
- [[后续概念1]] — 理解本页面后可以探索的概念
- [[后续概念2]]

## 原始来源

- [源文档显示名](raw/relative/path.md)
```

> `> [!abstract]`、`## 相关页面`、`## 原始来源` 是硬性要求。Frontmatter 中 title、date、tags 为必填。章节标题可灵活调整。

## 命名规范

- Wiki 页面用**中文**文件名，简洁描述性标题
- 概念页：`渲染管线.md`、`所有权与借用.md`、`闭包.md`、`Linux 进程模型.md`
- 实体页：`场景资源.md`、`Camera 组件.md`、`systemd 服务管理.md`
- 对比页：`Cocos Creator vs Unity.md`、`Rust vs C++.md`
- 概述页：`XXX 概述.md`（如 `Cocos Creator 概述.md`、`Rust 概述.md`、`Linux 概述.md`）
- 实践页：动宾短语，如 `实现角色控制器.md`、`配置 Arch Linux 开发环境.md`
- 特殊文件：`index.md`、`log.md` 固定小写英文
- 避免文件名中的 `/`、`\`、`:` 和多余空格

## 知识图谱策略

Wiki 的价值在于**连接**，不在于页面数量。

### 核心原则

1. **概念复用优于重复**：同一个概念只在一个页面深入解释，其他页面通过 wikilink 引用。例如，「异步编程」的通用原理在 `#layer/sw-eng` 中解释，JS/TS/Rust 各语言页面只解释语言特定的异步机制并链接回通用概念。

2. **Hub 页面**：每个领域创建一个概述页（`#type/overview`）作为概念枢纽，链接到该领域的所有核心概念页和上下层入口。概述页不是摘要——它是该领域的导航地图和概念索引。

3. **跨域桥梁**：识别跨领域的概念共鸣并显式建链：
   - **内存管理**：Linux（虚拟内存/页表）→ Rust（所有权/借用）→ JS（垃圾回收）——核心问题相同，表述不同
   - **并发模型**：Linux（进程/线程调度）→ Rust（Send/Sync/消息传递）→ JS（事件循环/微任务）——形成跨链共鸣
   - **模块化**：Linux（内核模块）→ Rust（crate 系统）→ JS（ES6 模块）→ TS（命名空间/模块解析）——概念演进链

4. **冲突即价值**：当不同源文档对同一概念给出矛盾结论时，显式记录冲突（`> [!conflict]` callout），标注 `#status/review`。冲突页面是高价值页面——它们标记了值得深入探索的知识边界。

### 反向链接检查

每次 Ingest 后，检查新修改的页面：
- 文中被提及但无 wikilink 的概念？→ 创建链接或标记待创建
- 已有页面应该链接到本页但没有？→ 更新那些页面
- 新页面只链接了同领域页面？→ 考虑是否有跨领域连接

## Ingest 工作流

处理 raw/ 中的源文档：

1. **读取**源文档全文
2. **识别知识归属**：
   - 这篇文档属于哪个领域层？
   - 它为现有概念增加了什么新视角？
   - 它能从其他领域的已有页面中获益吗？（如 Rust 的宏系统可从 sw-eng 的「元编程」概念中获益）
3. **识别影响**：
   - 增加了什么新知识？与已有主张有矛盾吗？
   - 有哪些关联概念已经存在 Wiki 页面？
4. **与用户讨论**关键发现和重点（除非是明确的批量 Ingest）
5. **更新/创建页面**：
   - 优先**深化现有页面**，而非创建新页面
   - **一个源文档 ≠ 一个 Wiki 页面**。好的 Ingest 可能创建一个新页面 + 更新 5-10 个已有页面
   - 如果创建新页面，确保它和至少 3 个已有页面建立 wikilink
   - 如果新知识与已有主张矛盾，添加 `> [!conflict]` callout 标注双方来源，设置 `#status/review`
   - 如果一篇源文档只够写出 3 段摘要 → 把内容合并到已有的相关页面中，不要创建孤立新页面
6. **更新交叉引用**：遍历所有被触动的页面，补充双向链接（含跨链链接）
7. **更新 `index.md`**：添加新页面到对应学习层分类表格
8. **追加 `log.md`**：`## [日期] ingest | 源文档标题` + 简要说明 + 受影响页面列表
9. **验证 raw/ 链接**：

```bash
grep -hPo '\(raw/[^)]+\)' wiki/<新页面> \
  | sed 's/[()]//g' | while read f; do \
    [ -f "$f" ] && echo "OK: $f" || echo "DEAD: $f"; \
  done
```

> 部分 raw/ 子目录的主文件不是 `index.md`（如 `raw/physics-2d/` 的主文件是 `physics-2d.md`）。引用前先 `ls` 确认实际文件名。

## Query 工作流

1. **使用 qmd 搜索定位相关页面**：
   - `qmd search "<关键词>" -c llm-wiki` — BM25 关键词搜索（快速，中文友好）
   - `qmd vsearch "<概念描述>" -c llm-wiki` — 向量语义搜索（适合概念性/模糊查询）
   - 如果搜索结果中 wiki/ 和 raw/ 混在一起，优先查看 wiki/ 路径的结果（已加工的 Wiki 页面）；raw/ 用于验证原始来源
2. 读 `index.md` 确认页面间的层级关系
3. 用 `qmd get "<qmd-uri>"` 读取 wiki 页面具体内容（URI 中文件名空格会转为 `-`，如 `wiki/并发模型对比` → `qmd://llm-wiki/wiki/并发模型对比`）
4. 按需读取相关 raw/ 源文档（用 Read 工具直接从文件系统读取）
5. 合成回答，附引用（页面名 + 源文档路径）
6. 如果回答中包含有价值的分析、比较或新发现（尤其是跨领域综合）——**问用户是否归档**

### qmd 搜索技巧

| 场景 | 命令 | 说明 |
|------|------|------|
| 精确概念定位 | `qmd search "所有权 借用" -c llm-wiki` | BM25，多关键词交集，返回高分匹配 |
| 语义/模糊查询 | `qmd vsearch "为什么 Rust 编译慢" -c llm-wiki` | 向量相似度，适合自然语言问题 |
| 带排除搜索 | `qmd search "材质 -2d" -c llm-wiki` | `-` 前缀排除词 |
| 混合精确搜索 | `qmd search '"事件循环"' -c llm-wiki` | 双引号表示短语精确匹配 |
| 查看文件内容 | `qmd get "qmd://llm-wiki/wiki/图形渲染"` | 快速预览文件内容（带行号） |

> **注意**：`qmd query` 在 CPU 模式下会超时（需加载 LLM 扩展模型），当前环境优先使用 `qmd search` 和 `qmd vsearch`。
> 运行 `qmd embed` 可补充 2432 个待嵌入文档以提升向量搜索质量。

### 检索决策

| 场景 | 首选入口 | 说明 |
|------|----------|------|
| 明确页面或领域 | `index.md` → Read | 目录已经承载当前概念图，避免过度搜索 |
| 中文关键词明确 | `qmd search` 或 Grep | BM25/文本搜索适合术语、文件名、标题 |
| 问题很模糊或跨领域 | `qmd vsearch` → `index.md` 校准 | 先语义召回，再用目录确认层级和关联 |
| 需要事实校验 | wiki 页面 → raw/ 来源 | wiki 给结构，raw/ 给可追溯证据 |
| 生成可复用成果 | 先答复，再询问归档 | 对比表、排障流程、综合分析适合沉淀为页面 |

> 不要把 qmd 当作传统 RAG 的替代品来绕过 wiki。qmd 是定位工具；最终回答应建立在已编译的 wiki 页面和可追溯 raw/ 来源之上。

### Query 归档原则

值得归档的回答类型：
- 跨多个页面的综合分析和连接
- 两个概念/系统/版本的对比
- 实际问题的解决方案或排查思路
- 新发现的概念关联或矛盾（尤其是跨链发现）

归档方式：
- 有现成页面可延展 → 追加章节
- 全新的综合 → 新建页面（标注 `#status/draft`）
- 追加 `log.md`：`## [日期] query | 归档主题`

归档判断：

| 问题类型 | 默认动作 |
|----------|----------|
| 一次性事实查询 | 只回答，不归档 |
| 概念解释且已有页面覆盖 | 回答并引用页面；如发现缺口，询问是否补充 |
| 多页面综合/对比 | 回答后询问是否归档为 `#type/comparison` 或补充到概述页 |
| 实战排障/操作流程 | 回答后询问是否归档为 `#type/practice` |
| 发现矛盾或过时内容 | 标记相关页面为 `#status/review`，添加 `> [!conflict]` 或待办说明 |

## Lint 检查

定期对 wiki 做健康检查。

### 结构检查

- 页面间矛盾声明（含 `> [!conflict]` callout 且未解决）
- 过时内容（与新版源文档对照，关注带版本标签的页面）
- 孤儿页面（零入站链接，排除 `#type/overview` 入口页）
- 遗漏的交叉引用（重要概念在文中被提及但没有 wikilink）
- 死链接（raw/ 引用和 wikilink）

全量死链检查：

```bash
grep -rhoP '\(raw/[^)]+\)' wiki/*.md \
  | sed 's/[()]//g' | sort -u | while read f; do \
    [ -f "$f" ] || echo "DEAD: $f"; \
  done
```

### 概念图检查

- **学习路径断链**：上层页面是否链接到必要的前提概念？
  - Cocos 页面 → 是否链接了所需的 JS/TS 前提？
  - Rust 页面 → 是否链接了所需的 Linux/系统编程前提？
  - TS 页面 → 是否链接了所需的 JS 前提？
- **跨链孤岛**：Rust 和 JS/TS 概念群之间应建立但未建立的连接？（如并发模型对比）
- **浅薄页面**：正文（不含 frontmatter、摘要、相关页面、原始来源）不足 ~500 字符 → 合并或扩展
- **重复概念**：同一概念在多个页面中以相似方式浅层解释 → 合并到一个概念页，其他页面用 wikilink 引用

### 时效性检查

- 标记 `#status/draft` 超过 30 天的页面 → 标记 `#status/review`
- `raw/` 源文档有更新（通过 `jj diff` 检测）→ 标记引用该文档的 Wiki 页面为 `#status/review`
- 带版本标签的页面 → 检查是否有新版本文档可用

### 标签合规检查

- 每页必须有至少一个 `#layer/`、一个 `#type/`、一个 `#status/` 标签
- 识别裸标签页面并迁移到带前缀格式
- 验证 `#layer/` 标签与页面实际所属领域一致

## 回答原则

- **第一性原理**：追溯到最基础的概念和原理来解释。回答"是什么"时，说明它解决的根本问题；回答"怎么做"时，拆解到底层机制。
- **费曼学习法**：用通俗易懂的语言解释，假设对方没有该领域背景。避免术语堆砌，优先用简单的比喻和具体的例子说明复杂概念。

## Index 与 Log 设计

### index.md

内容导向的 Wiki 导航文件。按学习层（由底向上）组织：

1. Linux / OS 基础
2. JavaScript 语言
3. TypeScript 类型系统
4. Rust 程序设计
5. Cocos Creator 引擎
6. 游戏开发实践
7. 软件工程（横向层）
8. 工具与参考

每层内部按页面类型分组（概述 → 概念 → 实体 → 对比 → 实践）。每个条目包含 `[[wikilink]]`、一句话摘要、可选的分类标签。

每次 Ingest 后必须更新 index.md。LLM 查询时先读取 index.md 定位相关页面。

### log.md

时间导向的操作记录。追加写入，不可修改历史条目。

格式：`## [YYYY-MM-DD] 操作类型 | 主题`

操作类型：`ingest` / `query` / `lint` / `restructure` / `setup`

每个条目包含：操作描述、受影响页面列表、关键发现或待办。

`log.md` 本身也是 `#type/meta` 页面，应包含 frontmatter 和摘要，但历史日志正文只追加新条目，不为美化而重排旧记录。

```bash
# 快速查看最近 5 条操作
grep "^## \[" log.md | tail -5
```
