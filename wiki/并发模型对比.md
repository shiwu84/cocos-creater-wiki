# 并发模型对比

> [!abstract] 摘要
> 并发模型是不同语言和运行时解决同一个根本问题的不同答案：**如何在多个独立任务之间安全地共享 CPU 时间与内存？** 从 Linux 的 OS 级进程隔离，到 JavaScript 的单线程事件循环，再到 Rust 的编译期并发安全验证，以及 Cocos Creator 的帧同步更新——每个模型在并行度、数据竞争防护、编程复杂度和运行时开销之间做出了不同取舍。本页将这四种并发模型并置比较，揭示它们之间的深层共鸣与本质分歧。

## 根本问题：多任务、单 CPU、共享内存

所有并发模型最终都在回答三个问题：

1. **调度**：谁来决定哪个任务何时运行？
2. **隔离**：任务之间如何保护彼此的数据不被意外修改？
3. **通信**：需要协作时，任务之间如何安全地传递信息？

不同模型对这三个问题的回答，构成了从"完全 OS 托管"到"完全编译期验证"的光谱。

## 四种并发模型

### Linux 进程/线程模型：OS 级抢占式并发

Linux 代表了**最底层、最通用**的并发模型。内核调度器以时间片轮转的方式在可运行进程（线程）之间切换，每个进程拥有独立的虚拟地址空间。

**调度**：由内核 CFS 调度器全权控制。进程无法预测自己被挂起的精确时刻——这是**抢占式**调度的本质：内核可以在任意指令边界中断当前进程。

**隔离**：进程通过虚拟内存实现强隔离——进程 A 无法访问进程 B 的内存，即使进程 B 野指针乱飞也影响不到进程 A。线程共享地址空间，隔离性弱，需要程序员手动使用互斥锁（mutex）等同步原语。

**通信**：进程间通过管道、信号、共享内存、Socket 进行通信。信号是异步通知机制（类似 OS 级的 Observer 模式），管道是流式数据通道。

| 维度 | Linux 进程 | Linux 线程 |
|------|-----------|-----------|
| 地址空间 | 独立，强隔离 | 共享，弱隔离 |
| 创建开销 | 大（fork + exec） | 较小（只分配栈） |
| 通信方式 | IPC（管道、信号、共享内存） | 直接读写共享变量 |
| 崩溃影响 | 隔离，不波及其他进程 | 可能导致整个进程崩溃 |
| 数据竞争 | 不存在（进程间） | 存在，需手动同步 |

> [!note] 抢占式 vs 协作式
> Linux 的抢占式调度意味着一个死循环不会阻塞整个系统——调度器会在时间片耗尽后强制切走。这与 JS 事件循环形成鲜明对比：JS 中一个 `while(true)` 会永久冻结整个页面。

### JavaScript 事件循环：单线程 + 异步回调

JavaScript 通过**放弃并行来换取安全**——整个运行时只有一个主线程，同一时刻只有一段代码在执行。

**调度**：由事件循环（Event Loop）驱动。任务分为宏任务（`setTimeout`、I/O）和微任务（Promise `.then/catch`、`queueMicrotask`）。每个宏任务执行完毕后，清空所有微任务队列，然后取下一个宏任务。详见 [[JavaScript Promise 与异步]]。

**隔离**：不需要隔离——因为同一时刻只有一个函数在执行，不存在两个线程同时修改同一个变量的可能性。数据竞争在 JS 主线程中**不存在**。

**通信**：异步操作通过回调、Promise、async/await 传递结果。Worker 线程通过 `postMessage` 进行结构化克隆通信（不能共享内存）。

```
执行栈清空 → 取一个宏任务执行 → 清空所有微任务 → 渲染更新 → 取下一个宏任务 → ...
```

> [!warning] 单线程的代价
> CPU 密集型任务（大量计算、图像处理）会阻塞事件循环，导致页面冻结无响应。解决方案是 Web Worker（真正的 OS 线程）或 WASM——但这意味着 JS 主线程的"无数据竞争"承诺仅限于主线程内部。

**微任务 vs 宏任务的优先级**体现了调度策略：

| 队列 | 来源 | 优先级 | 示例 |
|------|------|--------|------|
| 微任务 | Promise、MutationObserver | **高**——每个宏任务后立即清空 | `.then()`、`queueMicrotask()` |
| 宏任务 | setTimeout、I/O、事件 | 低——每次循环取一个 | `setTimeout(fn, 0)` |

### Rust 无畏并发：编译期验证 + 多范式

Rust 的并发模型是**对传统多线程并发的编译期安全增强**。它不限制你使用线程、共享内存或消息传递——但编译器会在编译时证明你的代码不存在数据竞争。

**调度**：由 OS 调度器管理（1:1 线程模型）。Rust 标准库不介入调度，但异步运行时（如 tokio）提供了 M:N 协程调度。

**隔离**：所有权系统提供了三种隔离范式：
1. **线程 + move**：将数据的所有权移入线程闭包，原线程无法再访问——编译期保证的线程隔离
2. **消息传递**（mpsc channel）：`send()` 移交所有权，发送后无法再使用——"通过通信来共享内存"
3. **共享状态**（`Arc<Mutex<T>>`）：类型系统强制你调用 `lock()` 才能访问内部数据——不能"忘记"加锁

**通信**：channel（发送即移交所有权）、`Arc<Mutex<T>>`（原子引用计数 + 互斥锁）、`std::sync::atomic`（无锁原子操作）。

> [!tip] 关键设计：Send 和 Sync trait
> `Send`：类型可以安全地将所有权转移到另一个线程。几乎所有 Rust 类型都实现了 `Send`（除了 `Rc<T>`）。
> `Sync`：类型的共享引用可以安全地在多线程间使用。`Mutex<T>` 实现了 `Sync`，但 `RefCell<T>` 没有。
> 这两个 trait 是编译器自动推导的——你组合出安全类型，编译器就自动验证线程安全性。详见 [[Rust 并发模型]]。

### Cocos Creator 更新循环：帧同步模型

Cocos Creator 的"并发"模型本质上是**时间分片而非真正的并行**——引擎在每个渲染帧内按固定顺序调用组件的生命周期回调。

**调度**：由引擎的主循环驱动，每帧依次执行：`update` → `lateUpdate` → 物理模拟 → 渲染。所有脚本代码都在主线程执行。

**隔离**：组件之间通过事件系统（EventTarget）通信，不直接"并发"访问彼此的数据。引擎提供了 `scheduleOnce`、`Tween` 等延迟执行机制，但都是**在主线程的时间线上排队**，不是创建新线程。

**通信**：`node.on()` / `node.emit()` 事件系统（Observer 模式）、`director.on()` 全局事件、`schedule()` 定时回调。

> [!note] 游戏引擎的"伪并发"
> Cocos 的更新循环与 JS 的事件循环共享同一个主线程——Update 回调本质上是被引擎主循环驱动的同步函数调用序列。你在 `update()` 里写死循环，整个游戏就卡死了。这与 JS 事件循环的"不能阻塞"约束如出一辙。

## 对比表格

| 维度 | Linux 进程/线程 | JavaScript 事件循环 | Rust 并发 | Cocos 更新循环 |
|------|----------------|---------------------|-----------|---------------|
| **并行性** | 真正并行（多核） | 单线程无并行（Worker 除外） | 真正并行（多核） | 单线程伪并行（帧分片） |
| **数据竞争防护** | 进程隔离（强）/ 手动同步（弱） | 无数据竞争（单线程） | **编译期杜绝数据竞争** | 无数据竞争（单线程） |
| **调度方式** | OS 抢占式（CFS） | 事件循环协作式 | OS 抢占式 + 异步运行时 | 引擎帧循环协作式 |
| **编程复杂度** | 中（需理解进程模型） | **低**（无锁编程） | 高（所有权 + 生命周期） | 低（回调式） |
| **CPU 密集型** | 优秀（充分利用多核） | **差**（阻塞主线程） | 优秀（安全多线程） | 差（与事件循环共享） |
| **I/O 密集型** | 中（线程池 / epoll） | **优秀**（异步非阻塞） | 优秀（tokio 等异步运行时） | 中（依赖 JS 异步） |
| **开销** | 中~高（上下文切换） | 低（无锁开销） | 零（编译期） + 运行时锁开销 | 低（单线程） |
| **典型场景** | 系统服务、高性能服务端 | Web 交互、Node.js IO | 系统编程、高性能计算 | 游戏逻辑、实时渲染 |

## 跨模型共鸣

### 协作式 vs 抢占式调度

并发模型的调度策略形成了两极：

- **抢占式**（Linux、Rust 线程）：OS 可以在任意时刻中断当前任务，无论它是否自愿让出 CPU。优点是不怕死循环，代价是上下文切换开销和同步复杂性。
- **协作式**（JS 事件循环、Cocos 更新循环）：任务必须主动返回控制权（结束当前函数、`await` 暂停），调度器才能切换到下一个任务。优点是编程简单、无抢占开销，代价是一个任务阻塞就会卡死全部。

### 隔离机制的三层光谱

```
            ┌──────────────────────────────────────┐
            │  Linux 进程：虚拟内存硬件隔离 (OS 层)   │  ← 最强隔离，最重开销
            │  Rust Send/Sync：编译期静态验证          │  ← 零开销隔离
            │  JS/Cocos 单线程：放弃并行 (架构层)      │  ← 无隔离成本，但无并行
            └──────────────────────────────────────┘
```

这三种隔离策略代表了安全性的三个不同来源：**硬件**（MMU 虚拟内存）、**编译器**（静态分析）、**架构决策**（单线程设计）。

### 消息传递的跨域实现

Actor 模型的思想——"不共享状态，通过消息通信"——在四个域中都有体现：

| 域 | 实现 | 机制 |
|----|------|------|
| Linux | 管道 / 信号 | 内核缓冲区 / 软件中断 |
| JS | `postMessage`（Worker）/ Promise | 结构化克隆 / 微任务队列 |
| Rust | `mpsc::channel` | 所有权移交（send 后无法再访问） |
| Cocos | `node.emit()` / `director.on()` | 事件总线 |

> 同一个模式，Linux 用内核管道实现，JS 用任务队列实现，Rust 用所有权实现，Cocos 用事件系统实现。理解模式本质后，切换技术栈时你只需要学习新的语法壳。

## 选型建议

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| Web 交互、Node.js 服务 | JS 事件循环 | 非阻塞 I/O 天然优势，单线程简化编程 |
| 高性能计算、多核利用 | Rust 线程 / Linux 原生线程 | 编译器保证安全，充分利用并行 |
| 系统守护进程 | Linux 进程 + fork | 进程隔离保证崩溃不扩散 |
| 游戏逻辑 | Cocos 帧循环 + JS 异步 | 帧同步适合确定性逻辑，异步适合 I/O |
| 需要零开销安全的系统编程 | Rust | 只有 Rust 能在编译期杜绝数据竞争 |
