# 模块化对比

> [!abstract] 摘要
> 模块化是所有软件系统增长到一定规模后的必然选择——它解决的根本问题是"如何将庞大的代码库分解为可独立理解、开发、测试和复用的单元"。不同生态系统对这个问题的回答在形式上高度同构，但在约束性和安全性上存在光谱式差异：Rust 的模块系统最严格（默认私有 + 显式 mod 声明），TypeScript/JS 最灵活（文件即模块 + 默认公开），Linux 的可加载内核模块 LKM 最特殊（运行在内核态但动态加载）。模块化不是单一技术——它是一种设计约束，不同的权限边界对应不同的系统可靠性需求。

## 一条渐变光谱

可视化：模块化系统的约束严格度：

```
更严格                                                更灵活
Rust ──── Linux LKM ──── TypeScript ──── JavaScript
│              │                │               │
默认私有       签名验证         默认公开        ES6 导入
mod 显式声明   固定接口         文件即模块      无类型约束
trait 为接口   内核态隔离       类型为接口      动态导入
```

## Rust：编译期隔离

Rust 的模块系统是最**结构化**的——它要求程序员显式声明模块树：

- `mod` 显式声明模块：模块不是自动的——需要在父模块中显式写 `mod my_module;`
- 默认私有：所有项（函数、结构体、字段）默认对父模块外不可见
- `pub` 逐级公开：`pub fn` 公开函数，`pub struct` 公开结构体但字段仍默认私有，需要 `pub` 逐字段公开
- crate 边界：库只能通过公开 API 被下游使用；二进制 crate 不可被依赖
- `use` 导入路径：`crate::module::item`——以 crate 根为起点的绝对路径

哲学：**"深思熟虑之后再公开"**——Rust 假设大多数代码是私有的，开发者需要主动决定哪些成为 API，因为一旦 `pub` 出去就是永久契约。

## TypeScript / JavaScript：文件级灵活

TS 沿袭了 JS 的模块约定但添加了类型层：

- 文件即模块：包含 `import`/`export` 的文件自动成为模块（不需要 `mod` 声明）
- 默认公开：`export` 出去的内容公开，未 `export` 的默认在模块内私有
- 类型空间：`import type` 只在编译期存在，不影响运行时
- 路径灵活：`import x from './foo'`、`import x from 'package'`——路径或包名都可以
- 模块解析可选策略：`node`（标准）、`classic`（历史）、`bundler`（打包工具）

关键差异：JS/TS 不需要显式声明模块树——文件系统的目录结构**就是**模块树。这在小型项目中极其便利，但在大型项目中（数百个文件）容易丢失全局结构。

## Linux LKM：内核空间的特殊案例

Linux 可加载内核模块提供了一种极端的模块化——运行在内核态，动态加载/卸载：

- 隔离方式：内核地址空间（单个模块崩溃可能导致整个内核崩溃——没有进程隔离）
- 接口：通过 `EXPORT_SYMBOL` 宏导出符号供其他模块调用
- 签名验证：`CONFIG_MODULE_SIG` 要求模块必须用可信密钥签名
- 动态加载/卸载：`insmod`/`rmmod`/`modprobe`——系统运行时无需重启
- 生命周期：`module_init` / `module_exit` 回调

LKM 是"没有进程隔离的模块化"——所有内核模块共享同一个地址空间，信任关系必须通过签名验证而不是内存隔离来保证。

## 对比表

| 维度 | Rust crate | TS/ES 模块 | Linux LKM |
|------|-----------|-----------|-----------|
| **模块建立方式** | `mod` 显式声明 | 文件 `import`/`export` 即模块 | 编译为 `.ko` + `insmod` |
| **默认可见性** | 私有 | 模块内私有（需要 export） | 模块内私有（需要 EXPORT_SYMBOL） |
| **外部依赖发现** | `Cargo.toml` + crates.io | `package.json` + npm registry | 无集中注册，发行版仓库 |
| **版本管理** | 语义化版本 + Cargo.lock | 语义化版本 + package-lock.json | 内核版本匹配（modver） |
| **隐私性强度** | 很强（逐字段 pub） | 中等（类型信息可被路径推断） | 弱（所有代码在同地址空间） |
| **热替换** | 不支持（编译期） | HMR（开发服务器支持） | 支持（rmmod + insmod） |
| **类型安全** | 编译期完整类型检查 | 编译期类型检查（TS）/ 无（JS） | 无（C 语言，手动） |

## 跨层洞察

### 为什么 Rust 最严格？

因为 Rust 在设计哲学上追求"如果一个模块能运行，那么它大概率是正确的"。默认私有 + 显式 `pub` 是在保护 API 设计——迫使库作者仔细思考哪些应该是公开 API，而不是"所有函数默认公开"。

### 为什么 JS 最灵活？

因为 JS 的历史包袱：在没有模块系统的时代（script 标签），所有变量在全局作用域。ES6 模块是后来添加的，且必须兼容庞大的 npm 生态（50 万+ 包）。模块化是"外部加入"的，而非"内建"的。

### Linux LKM 为什么特殊？

因为它是唯一运行在**内核态**的模块化系统——没有 OS 层的保护。传统模块化（用户态）的"模块崩了不影响其他"在 LKM 中不成立——一个 LKM 的 bug 可能导致整个内核崩溃。

## 与软件工程的概念共鸣

模块化是软件工程中最基础的横向概念——它同时出现在：
- **代码组织**层（源码如何拆分为文件和目录）
- **依赖管理**层（外部依赖如何声明和解析）
- **运行时加载**层（模块何时装入地址空间并开始执行）
- **API 设计**层（模块暴露出什么作为使用契约）

任何大型系统都必须在这些层面做出选择，Rust/TS/Linux 代表了三条不同的选择路径，但它们在根本上回答的是同一个问题。
