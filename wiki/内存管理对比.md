# 内存管理对比

> [!abstract] 摘要
> 内存管理是所有程序都必须面对的根本问题——如何在有限的内存中高效、安全地为不断变化的程序需求服务。不同语言和运行时对这个问题的回答形成了一个光谱：Linux（OS 级）通过虚拟内存和页表为进程提供隔离的地址空间；Rust（编译期）通过所有权和借用在编译时消除内存错误；JavaScript（运行时）通过标记-清除 GC 自动回收不可达对象。这三个层次不是竞争关系，而是分层协作——V8 的 GC 运行在 Linux 虚拟地址空间之上，Rust 的所有权检查生成的操作系统调用最终通过虚拟内存实现安全。理解这些层次如何协作，才能真正理解"为什么程序不会一次分配就耗尽系统内存"。

## 三个层级

```
┌──────────────────────────────┐
│  JavaScript 运行时（V8）      │  ← 标记-清除 GC：自动回收不可达对象
│  • 新生代/老年代分代           │     "程序不关心何时释放，运行时猜测"
│  • Stop-the-world + 增量      │
├──────────────────────────────┤
│  Rust 编译期                 │  ← 所有权 + 借用：编译器验证每次内存操作
│  • 编译期确定释放点            │     "释放是确定的，编译器帮你记着"
│  • Rc/Arc 推迟到运行时        │
├──────────────────────────────┤
│  Linux 内核（OS 层）          │  ← 虚拟内存 + 页表：进程隔离地址空间
│  • 每个进程有独立虚拟空间       │     "进程只能看到自己的内存映射"
│  • 物理内存由内核统一管理       │
└──────────────────────────────┘
```

## Linux：虚拟内存与页表

Linux 提供的是**基础设施级**的内存管理——它不关心你的程序是用 Rust 还是 JS 写的，它只提供：

- **虚拟地址空间**：每个进程看到的是从 `0x0` 开始的连续地址，实际物理地址由页表映射。这意味着进程 A 的 `0x1000` 和进程 B 的 `0x1000` 指向完全不同的物理内存。
- **页表**：虚拟地址到物理地址的映射表。内核在上下文切换时切换页表根指针（`CR3` 寄存器）。
- **按需分配**：`malloc` 只是修改进程的地址空间记录（VMA），物理内存直到第一次写入时才真正分配（页错误→内核分配物理页）。
- **交换**：物理内存不足时，不活跃的页面被写到 swap 分区/文件，释放物理内存。

| 概念 | Linux 实现 | 目的 |
|------|-----------|------|
| 隔离 | 独立虚拟地址空间 | 进程 A 不能读写进程 B 的内存 |
| 超量分配 | 按需分配 + swap | 程序申请的内存可以大于物理内存 |
| 共享 | 共享映射/写时复制 | 多个进程共享同一物理内存（如共享库、fork 后的父子进程） |

## Rust：编译期所有权

Rust 在 Linux 提供的虚拟内存基础设施之上，添加了**编译期**的安全保证。Rust 不改变分配和释放的底层机制（`malloc` 仍是 `malloc`，`free` 仍是 `free`），但它保证每一次 `free` 都"正好一次"发生。

- **所有权**：每个值有且仅有一个所有者。所有者离开作用域时，值被释放。
- **借用规则**：同一时刻只能有一个可变引用或多个不可变引用——消除了编译期的数据竞争。
- **RAII**：资源获取即初始化——分配和释放被绑定到作用域，编译器自动插入释放代码。

| 操作 | Linux 态 | Rust 的额外保证 |
|------|---------|----------------|
| 分配堆内存 | `mmap`/`brk` 系统调用 | 编译器确保分配的值被绑定到所有者 |
| 释放 | `free`/`munmap` | 编译器确定释放时机——不会忘记，不会重复 |
| 空指针 | 解引用 → 段错误 | 编译器强制 `Option<T>` 检查 |

> [!note] 关键理解
> Rust 的所有权不是替代 Linux 的内存管理，而是**在 Linux 的分配/释放原语之上添加了一个编译器验证层**。`Box<T>` 的 `new` 最终调用 `malloc`，`Drop` 最终调用 `free`——只是 Rust 编译器在中间证明了"free 只会被调用一次且不会在调用后继续使用"。

## JavaScript：自动垃圾回收

JS 引擎（V8、SpiderMonkey）运行在 OS 虚拟内存之上，另外添加了一个**运行时 GC** 层。引用关系被运行时追踪，不可达对象被自动回收。

- **标记-清除**：从根对象（全局变量、栈变量）出发遍历所有可达对象，标记；未标记的视为垃圾，清除。
- **分代**：大多数对象"朝生暮死"——新生代频繁 GC（Scavenge），老年代偶尔 GC（Mark-Sweep/Mark-Compact）。
- **问题**：GC 暂停——标记阶段通常需要停住所有 JS 执行（stop-the-world）。

| 维度 | Rust 所有权 | JS GC |
|------|-----------|-------|
| 释放时机 | 编译期确定（离开作用域） | 运行时确定（GC 周期触发） |
| 释放代价 | 零运行时开销 | GC 暂停 |
| 内存占用 | 即时释放，紧凑 | 延迟释放，可能有碎片 |
| 编程负担 | 学习曲线高（所有权规则） | 低（不用思考释放） |
| 确定性 | 完全确定 | 不确定（取决于 GC 策略和堆状态） |

## 三个层次如何协作

以一个 Rust 程序中的字符串操作为例：

1. `let s = String::from("hello")` → Rust 编译器确定此分配的释放点在 `s` 离开作用域时
2. 底层调用 `malloc` → Linux 在虚拟地址空间中分配 VMA（但物理页可能还未分配）
3. 第一次写入 "hello" → 触发页错误 → Linux 分配物理页并建立页表映射
4. `s` 离开作用域 → `Drop::drop` 调用 `free` → Linux 释放该虚拟地址范围（物理页待回收）

在 JS 中：
1. `const s = "hello"` → 分配在 V8 堆上（V8 内部管理的内存池）
2. 如果 s 不再被引用 → 下一次 GC 周期标记-清除发现它不可达 → 释放回 V8 的内存池
3. V8 可能将大块空闲内存 `munmap` 返回给 OS，也可能保留用于未来分配

## 跨层核心洞察

1. **抽象层次不竞争而是分层**：Rust 的借用检查器不可能替代 Linux 的页表保护（因为进程间隔离需要硬件级机制），Linux 的页表也不可能替代 GC（因为用户态程序无法安全地追踪跨栈引用关系）。
2. **同构的问题，不同层次的答案**："一个对象何时不再被需要"——Linux 说"进程结束时"，Rust 说"所有者离开作用域时"，JS 说"没有引用指向它时"。
3. **GC 不是"自动化 malloc/free"**：GC 解决的是"可达性追踪"问题（跟踪对象引用图），而 Rust 的所有权解决的是"资源生命周期的形式化验证"问题。这两个问题不同，混淆它们会导致错误地认为"有了 GC 就不需要所有权"，或"有了所有权就不需要 GC"。
