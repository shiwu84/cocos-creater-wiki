# Rust 宏系统

> [!abstract] 摘要
> Rust 宏是实现元编程的方式——编写"生成代码的代码"。它解决的根本问题是"如何减少重复的样板代码，当普通的函数、泛型、trait 都做不到时"。宏在编译阶段展开为 Rust 源代码，因此它接收的是 token 流而非运行时值。Rust 提供两类宏：声明宏（`macro_rules!`）用模式匹配生成代码，过程宏（`#[derive]`/属性/类函数）用 Rust 代码操作 token 流。`println!`、`vec!`、`#[derive(Debug)]` 都是宏——它们做的是函数做不到的事。

## 根本问题：元编程

泛型让函数操作不同**类型**，但不能让函数接收**可变数量的参数**或在**编译时生成 trait 实现**。泛型的极限是函数签名的静态性——所有参数的类型和数量必须在签名中写死。宏消除了这个限制——它在"代码生成代码"的层面操作，发生在编译的前端。

真正需要宏的场景：
- **可变参数**：`println!("{} {}", a, b, c)`——函数做不到
- **编译期代码生成**：`#[derive(Debug)]` 自动实现 `Debug` trait——trait 不能在运行时实现
- **嵌入 DSL**：如 SQL 查询、HTML 模板——在编译时验证 DSL 语法
- **消除重复**：当 trait 和泛型不够时（如重复实现一组类似 trait）

## 声明宏：`macro_rules!`

声明宏的工作方式类似 `match` 表达式——将输入的 token 流与模式匹配，然后替换为对应的代码。

```rust
macro_rules! vec {
    ( $( $x:expr ),* ) => {
        {
            let mut temp_vec = Vec::new();
            $(
                temp_vec.push($x);
            )*
            temp_vec
        }
    };
}
```

核心语法：
- `$( ... )` 捕获重复模式
- `$x:expr` 匹配任意 Rust 表达式并命名为 `$x`
- `*` 表示零次或多次重复
- `#[macro_export]` 使宏在 crate 外部可见

声明宏适合简单的"模式→替换"场景。复杂的宏逻辑应该用过程宏。

## 过程宏：三种形式

过程宏是 Rust 函数，输入 `TokenStream`，输出 `TokenStream`。它们必须定义在独立的 `proc-macro` 类型的 crate 中。

### 自定义派生宏（`#[derive(...)]`）

最常见的宏类型。在结构体/枚举上自动生成 trait 实现：

```rust
#[derive(HelloMacro)]
struct Pancakes;
```

编译器将 `Pancakes` 的定义作为 token 流传给宏函数，宏函数生成 trait impl 代码返回。

### 类属性宏

类似 derive 宏，但可用于任意项（函数、模块等），而不仅仅是类型定义。如异步框架的 `#[tokio::main]`。

### 类函数宏

调用形式像函数但更灵活：`sql!(SELECT * FROM users)`。与声明宏的区别是操作的是 token 流而非模式匹配。

## 宏 vs 函数

| 维度 | 宏 | 函数 |
|------|-----|------|
| 参数 | 可变数量，任何 token | 固定数量，具体类型 |
| 执行时机 | 编译时展开 | 运行时 |
| 定义位置 | 调用前必须先定义/导入 | 任意顺序 |
| 复杂性 | 难读、难调试 | 清晰 |

## 跨语言对比

| 语言 | 宏/元编程 | 特点 |
|------|----------|------|
| Rust | `macro_rules!` + 过程宏 | 卫生宏，编译时展开，不需要预处理阶段 |
| C/C++ | `#define` 预处理宏 | 非卫生（可能的名冲突），简单文本替换 |
| Lisp | `defmacro` | 最强大的宏系统，代码即数据 |
| JS/TS | 无原生宏，靠 Babel 插件 | 生态外挂，非语言特性 |
| TS | 类型体操（template literal types 等） | 某种程度上是"类型级宏" |

Rust 宏是**卫生宏**（hygienic）：宏内引入的名字不会意外与宏外名字冲突，避免了 C 预处理器的一大痛疾。

## 注意事项

- **宏的调试难度远高于函数**：错误信息指向展开后的代码，而非宏定义本身
- **不要用宏做函数能做的事**：如果泛型或 trait 能解决，用宏是过度设计
- **过程宏必须自己处理 `syn`/`quote` 依赖**：它们不是标准库的一部分
- **过程宏的编译开销**：每个过程宏 crate 都需要单独编译
