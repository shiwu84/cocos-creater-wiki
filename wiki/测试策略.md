# 测试策略

> [!abstract] 摘要
> 测试解决的根本问题是**变更的信心**：当你修改代码后，你有多大的信心确定自己没有破坏已有功能。测试金字塔（单元 → 集成 → E2E）描述了不同类型测试的投资回报比——底部（单元测试）运行快、定位精确、维护成本低；顶部（E2E）覆盖真实场景但慢且不稳定。Rust 将测试设施内建在语言和工具链中（`#[test]`、`cargo test`），JS 依赖生态拼装（Mocha/Jest/Vitest）。这一差异反映了两者不同的设计哲学：Rust 认为测试是"正确性保证"的延续，JS 认为测试是"开发者选择"的增强。

## 根本问题：代码正确性不可穷举证明

Dijkstra 的名言精确地定义了测试的边界："测试能证明 bug 的存在，但无法证明 bug 的不存在。"Rust 的类型系统和借用检查器消除了整个类别的运行时错误（空指针、数据竞争、内存泄漏），但它们无法验证**业务逻辑的正确性**——编译器不会知道 `add_two(3)` 是否应该返回 `5` 还是 `50`。

测试的作用不是"证明程序正确"，而是"通过自动化断言降低错误回归的概率"。每次 CI 通过，你获得的是"至少到目前为止没出错"的有限信心。

## 测试金字塔

```
      /\
     /E2E\      少量，验证完整用户流程，慢但覆盖关键路径
    /------\
   / 集成测试 \   中等量，验证模块间交互，中等速度
  /----------\
 /  单元测试    \  大量，验证独立函数/方法，极快
 --------------
```

- **单元测试**：验证单个函数/方法的纯逻辑。隔离依赖（mock/stub 外部服务）。运行毫秒级。
- **集成测试**：验证多个模块的交互。在 Rust 中，集成测试只能调用库 crate 的公共 API（模拟外部用户）。
- **E2E 测试**：模拟真实用户行为。维护成本高（UI 变化后易碎），应只覆盖核心流程。

核心原则：**越底层的测试越值得投入**，因为它们的维护成本最低，编写最快，反馈循环最短。如果一个问题可以被单元测试捕获，就不应该等待集成测试或 E2E 来发现。

## Rust 测试体系

### 语言内建设施

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_add_two() {
        assert_eq!(add_two(3), 5);
    }
}
```

- `#[test]` 标注测试函数
- `#[should_panic]` 标注期望 panic 的测试
- `assert!`、`assert_eq!`、`assert_ne!` 是标准库提供的断言宏
- 测试与源码可以写在同一文件中（`#[cfg(test)]` 模块只在测试时编译）

### 测试组织约定

```
src/lib.rs              ← 库代码 + 内联单元测试
tests/
  integration_test.rs   ← 集成测试（只能调用 crate 的公共 API）
```

集成测试位于 `tests/` 目录，每个 .rs 文件作为独立的 crate 编译。

### `cargo test`

`cargo test` 覆盖所有测试需求：
- `cargo test`：运行所有测试（单元 + 集成 + 文档测试）
- `cargo test test_name`：运行名称匹配的测试
- `cargo test -- --test-threads=1`：单线程运行（测试有共享状态时）
- `cargo test -- --ignored`：运行标记为 `#[ignore]` 的测试

Rust 的 `cargo test` 是"一切都装在一个工具里"哲学的又一流：不需要安装测试框架、测试运行器、断言库——这和 JS 生态形成鲜明对比。

## JS 测试生态

```javascript
// Mocha + Chai 示例
describe('addTwo', function() {
    it('should add two to a number', function() {
        assert.equal(addTwo(3), 5);
    });
});
```

JS 没有标准化测试框架，需要通过 npm 安装：
- **Mocha**：测试运行器 + 结构化描述（`describe`/`it`）
- **Chai**：断言库（`expect`/`assert`/`should`）
- **Jest/Vitest**：更现代的"全家桶"——包含运行器、断言、Mock、覆盖率

### 哲学差异

| 维度 | Rust | JavaScript |
|------|------|-----------|
| 测试设施 | 语言 + 工具链内建 | 社区生态拼装 |
| 框架选择 | 无需选择（一个标准） | 需要选择（Mocha/Jest/Vitest） |
| 测试位置约定 | 约定清晰（`#[cfg(test)]`、`tests/`） | 约定多样（`__tests__/`、`.test.ts`、`.spec.ts`） |
| 断言风格 | 宏（编译期检查） | 方法调用（运行时） |
| 哲学 | 测试是"正确性保证"的延续 | 测试是"开发者选择"的增强 |

Rust 的"一个标准答案"让新手入门时没有决策疲劳（不需要先选框架），但灵活性不如 JS。JS 的繁荣生态提供了不同范式的选择（如 Jest snapshot 测试在 Rust 中不存在），但决策成本更高。

## 实践原则

1. **先写单元测试**：如果一个 bug 可以在 <1ms 的单元测试中被捕获，不应该等到 3 分钟的 E2E 测试来发现
2. **测试私有逻辑？**：Rust 可以测试私有函数（通过 `use super::*` 在 `#[cfg(test)]` 模块中）；这在 JS 中更困难——这是好事：测试私有函数意味着实现细节泄漏到测试中
3. **Mock vs 真实**：尽可能使用真实对象；只有当依赖不可控（外部 API、数据库）时才 mock
4. **文档测试是 Rust 的特色**：`/// 代码示例` 既是文档也是测试（`cargo test` 会运行它们）

## 注意事项

- **测试覆盖率 ≠ 测试质量**：100% 覆盖的糟糕测试比 50% 覆盖的有意义测试更差——它给人一种虚假的信心
- **不要为测试而测试**：如果你对测试的正确性毫无信心（逻辑太复杂需要测试测试），说明实现应该被拆分
- **测试是代码的一部分**：测试也需要维护、重构、设计。避免复制粘贴测试（用参数化测试/table-driven tests）
