# 错误处理策略对比

> [!abstract] 摘要
> 错误处理不是语言的附属品——它反映了一门语言对"什么是错误、谁该为错误负责"的根本哲学立场。本文对比四种错误处理策略：Linux 的 OS 级信号通知、JavaScript 的异常抛出与 Promise 拒绝、Rust 的 Result 值类型与 panic 崩溃、Cocos Creator 的混合模型。核心分歧在于：**错误是特殊的控制流（异常），还是常规的返回值（Result）？** 处理是可选的（JS、Linux），还是被编译器强制要求的（Rust）？这决定了代码的健壮性底线和调试体验。

## 根本分歧：错误是什么？

编程语言对"错误"有两种根本不同的认知：

| 观点 | 代表语言 | 信任模型 |
|------|---------|---------|
| **错误是异常** | JS、Java、Python | 信任程序员：你知道什么时候该 catch，不强制 |
| **错误是值** | Rust、Go、Haskell | 不信任程序员：编译器强制你面对每个可能的失败 |

这场分歧的实质是：**是否信任程序员会记得处理每一个可能的错误？** Rust 的回答是"不"——所以用类型系统强制执行。JS 的回答是"信任，但提供工具"——所以 `try-catch` 是建议而非强制。

## 四种策略

### Linux 信号：OS 级的外部错误通知

Linux 信号机制代表了最底层的错误处理模型——**错误是从外部闯入进程的异步事件**。

**机制**：当硬件检测到非法操作（段错误、除零）或其他进程发送信号（SIGTERM、SIGKILL）时，内核向目标进程投递信号。进程可以注册信号处理器来响应。

**哲学**：信号不区分"可恢复错误"和"致命异常"——SIGSEGV（段错误）和 SIGTERM（请求终止）都是同一个机制。进程自己决定如何处理。

**关键设计**：
- SIGKILL (9) 和 SIGSTOP (19) **不可捕获**——这是操作系统的最后保障
- 信号是**异步**的：错误可能在任意指令边界到达，信号处理器运行在独立的信号栈上
- 信号处理器中能做的事情极其有限（信号安全函数限制）——因为不知道被打断时程序处于什么状态

| 信号 | 含义 | 可捕获？ | 典型处理 |
|------|------|---------|---------|
| SIGSEGV | 段错误（非法内存访问） | 可捕获 | debug 用，生产环境通常直接 core dump |
| SIGPIPE | 向关闭的管道写入 | 可捕获 | 忽略或优雅关闭连接 |
| SIGTERM | 请求终止 | 可捕获 | 优雅退出（关闭文件、释放锁） |
| SIGILL | 非法指令 | 可捕获 | 通常是硬件/编译器 bug，core dump |
| SIGBUS | 总线错误（未对齐访问） | 可捕获 | 硬件问题，core dump |

> [!note] 信号 vs 返回值
> Linux 的信号是**旁路**通知机制——错误通知与正常的函数调用-返回路径**不在同一条执行流上**。这使其适合处理"超出进程控制的意外事件"（如硬件异常、外部 kill），但不适合处理"函数未能完成预期任务"这种常规业务错误。对于常规错误，Linux 使用 `errno` + 返回值（C 风格）。

详见 [[Linux 进程模型]] 中关于信号的完整讨论。

### JavaScript try-catch + Promise rejection：双轨模型

JavaScript 有两套错误处理机制，各适用于不同的执行模型：

**同步代码**：`try...catch...finally`

```javascript
try {
    const data = JSON.parse(malformedString); // SyntaxError
} catch (err) {
    console.error(err.name, err.message);
} finally {
    cleanup();
}
```

**异步代码**：Promise rejection + `.catch()` / `try-catch` + `await`

```javascript
// Promise 风格
fetch(url)
    .then(res => res.json())
    .catch(err => console.error('Network error:', err));

// async/await 风格（本质是 Promise 的语法糖）
try {
    const res = await fetch(url);
    const data = await res.json();
} catch (err) {
    console.error(err);
}
```

**关键差异**：

| 特性 | try-catch（同步） | Promise rejection（异步） |
|------|-------------------|--------------------------|
| 作用域 | 包围的同步代码块 | 整条 Promise 链 |
| 跳过处理 | 可以（错误向上冒泡） | 可以被忽略（unhandledrejection） |
| 多层处理 | `catch` 内可 `rethrow` | `.catch()` 可返回 rejected Promise 传递 |
| 运行时行为 | 立即中断当前执行流 | 微任务队列处理 rejection |

> [!warning] 未处理的 Promise 拒绝
> JS 的双轨模型有一个危险的裂隙：Promise 被拒绝但没有 `.catch()` 处理器时，不会像同步异常那样自动中断程序。早期 Node.js 对此只打印警告（现已默认终止进程），浏览器触发 `unhandledrejection` 事件。这意味着**异步错误可能在你不知情的情况下被静默丢弃**——这是 JS 错误处理最容易被忽视的陷阱。

**Error 层次体系**：JS 提供了内建错误类型层次（`Error` → `SyntaxError`、`TypeError`、`ReferenceError`），支持 `instanceof` 分类型处理。详见 [[JavaScript 错误处理]]。

### Rust Result\<T, E\> + panic!：值与崩溃的分治

Rust 对错误处理做了最鲜明的区分：**可恢复的用 Result，不可恢复的用 panic**。

#### Result\<T, E\>：错误即值

```rust
use std::fs::File;

fn read_config() -> Result<String, std::io::Error> {
    let file = File::open("config.toml")?; // ? 运算符：成功则取出值，失败则提前返回 Err
    // ...
    Ok(content)
}
```

`?` 运算符是 Rust 错误处理的精髓：它让错误传播像异常一样简洁，但**类型签名明确告知调用者"这个函数可能失败"**。

**强制处理**：

```rust
let file = File::open("data.txt");
// file 的类型是 Result<File, Error>
// 你不能直接把它当 File 用——编译器强制你处理错误分支

// 必须显式处理：
let file = match File::open("data.txt") {
    Ok(f) => f,
    Err(e) => {
        eprintln!("Failed: {e}");
        return;
    }
};
```

你**无法不小心忽略错误**——类型系统在编译期就拦截了。

> [!tip] Result 与 `?` 的组合力量
> `?` 不是"吃掉错误"——它是把错误**沿着调用链向上传播**。当你在一串可能失败的操作后写 `?`，你实际上在说："如果任何一步失败，就把错误原样抛给上层调用者"。这种**显式的错误传播**让你清晰地看到错误在调用栈中的流动路径。

#### panic!：不可恢复的崩溃

```rust
panic!("unrecoverable state reached");
```

`panic!` 用于"不应该发生"的情况：数组越界（release 模式未检查）、断言失败、状态不一致。默认行为是展开栈（unwinding）并终止线程。

**何时用 Result，何时用 panic？**

| 场景 | 使用 | 理由 |
|------|------|------|
| 文件不存在 | `Result` | 可预期的外部故障 |
| 网络超时 | `Result` | 调用者可能想重试 |
| 输入格式错误 | `Result` | 用户输入，非程序 bug |
| 数组越界（release） | ~~panic~~ | 这是程序员 bug，但 Rust std 不强制检查 |
| 锁被毒化（poisoned） | `Result`（`PoisonError`） | 另一个线程 panic 导致的状态不一致 |
| 除零 | `panic!` | 数学无定义，逻辑错误 |

#### Option\<T\>：当"没有值"不是错误

对于"值可能不存在"的场景，Rust 使用 `Option<T>`（`Some` / `None`），将其与"操作失败"（`Result<T, E>`）区分开：

```rust
let first = vec![1, 2, 3].first(); // Option<&i32>
match first {
    Some(val) => println!("First: {val}"),
    None => println!("Empty vec"),
}
```

> 将"值不存在"（Option）和"操作失败"（Result）作为不同的类型，是 Rust 对错误语义的精确定义——它迫使你思考：这是一个可预期的空值，还是一个需要处理的错误？JS 中两者都通过 `null` / `undefined` / `throw` 混在一起表达。

### Cocos Creator：混合模型

Cocos Creator 的错误处理融合了 JS 的异常机制和引擎特有的事件通知系统：

**脚本层**：标准 JS `try-catch` + Promise rejection（因为 Cocos 脚本语言是 TypeScript/JS）。

**引擎层**：引擎通过事件系统通知错误和异常状态：

- 资源加载失败：`assetManager.loadRemote()` 的 Promise rejection，或在回调中检查 `err` 参数
- 引擎内部错误：通过 `director.on()` 或 `game.on()` 注册的引擎事件
- 编辑器扩展错误：Console 面板输出 + 编辑器日志

**实践模式**：

```typescript
// 资源加载：Promise 风格错误处理
assetManager.loadRemote<ImageAsset>(url, { ext: '.png' }, (err, asset) => {
    if (err) {
        console.error('Load failed:', err.message);
        return;
    }
    // 使用 asset
});

// 或 async/await 风格
try {
    const bundle = await assetManager.loadBundle('game');
} catch (err) {
    console.error('Bundle load failed:', err);
}
```

**特点**：

- 继承了 JS 错误处理的所有能力和所有陷阱（包括未处理的 Promise rejection）
- 引擎提供的事件系统弥补了 JS 异常模型在"全局状态异常通知"上的不足——引擎可以通过事件告知所有相关组件"发生了某种系统级问题"
- 资源加载是最高频的异步错误场景，Cocos 的回调式 API（`(err, asset) => {}`）本质上是 Node.js 的错误优先回调模式

## 对比表格

| 策略 | 可恢复错误 | 不可恢复错误 | 是否强制处理 | 错误类型信息 | 组合性 |
|------|-----------|-------------|-------------|-------------|--------|
| **Linux 信号** | 可捕获信号（SIGTERM） | 不可捕获信号（SIGKILL） | 否（信号可被忽略） | 信号编号，无结构化信息 | 差（异步，独立于调用栈） |
| **JS try-catch** | `throw new Error()` | 未捕获异常→脚本终止 | **否**（完全可选） | 有（Error 层次体系） | 中（try-catch 可嵌套） |
| **JS Promise** | `.catch()` / `try-catch await` | unhandledrejection | **否**（容易遗漏） | 有（错误对象传递） | 好（链式传播） |
| **Rust Result** | `Result<T, E>` + `?` | — | **是**（编译器强制） | 有（泛型 `E` + 自定义错误类型） | **优秀**（`?` 运算符） |
| **Rust panic!** | — | `panic!` → 栈展开 | 不强制（recover 需 `catch_unwind`） | 有（`&str` 消息 + backtrace） | 差（旁路机制） |
| **Cocos 混合** | try-catch + 回调 err | 引擎事件通知 | 否 | 有（JS Error + 引擎错误码） | 中（依赖 JS 机制） |

## 核心洞察

### 1. 类型系统是错误处理的分水岭

Rust 的 `Result<T, E>` 之所以能强制处理错误，根本上是因为**类型系统参与了错误表达**。`File::open` 的返回类型不是 `File`，而是 `Result<File, Error>`——你拿到的不直接是你要的东西，而是"可能的东西"。这种类型级区分迫使调用者面对失败路径。

JS 的 `try-catch` 无法做到这一点，因为 JS 的类型系统不参与错误表达——任何函数都可以随时 `throw`，而调用者从签名上根本看不出来。TypeScript 在这一点上也无能为力——它不标注函数可能抛出什么异常。

> [!conflict] 异常安全 vs 显式传播
> Rust 的 `Result` 模型写得更多但**读得更清楚**——你看到函数签名就知道它可能失败，看到 `?` 就知道错误在向上传播。JS 的 `try-catch` 写得少但**隐藏了错误路径**——不读函数体你无法知道它是否抛出异常。
>
> - **Rust 优势**：不会遗漏错误处理，错误传播路径显式可见
> - **JS 优势**：Happy path 代码更简洁（不被错误处理淹没）
> - **权衡**：Rust 初学时"样板代码多"的感觉，随着项目规模增长会变成"代码意图清晰"的体验

### 2. 可恢复 vs 不可恢复的边界定义

不同语言对"什么错误可以恢复"的定义差异，反映了其设计目标：

- **Linux**：信号模型不对错误分类——SIGSEGV（段错误）和 SIGTERM（礼貌终止）用同一机制。因为 OS 不知道进程的语义，它只能通知"发生了某事"。
- **JS**：几乎所有错误都可以 `catch`，甚至可以 `catch` 后吞掉（swallow）——这给了开发者最大灵活性，但也允许错误被静默隐藏。
- **Rust**：`Result` 用于预期失败（文件不存在、网络超时），`panic!` 用于不应发生的 bug（除零、断言失败）。这条边界由**程序员**而非语言决定——`unwrap()` 把 Result 的错误转为 panic，`catch_unwind` 把 panic 转为 Result。

### 3. 组合性是隐藏的竞争力

`Result` 的 `?` 运算符解决了错误处理中一个普遍痛点：**错误传播代码淹没业务逻辑**。对比：

```rust
// Rust：? 让错误传播几乎不可见
fn process() -> Result<Output, Error> {
    let config = read_config()?;      // 失败则提前返回
    let data = fetch_data(&config)?;  // 失败则提前返回
    let result = compute(data)?;      // 失败则提前返回
    Ok(result)
}
```

```javascript
// JS async/await：最近的等价形式，但异常路径不可见
async function process() {
    const config = await readConfig();    // 可能 throw
    const data = await fetchData(config); // 可能 throw
    const result = await compute(data);   // 可能 throw
    return result;
}
```

两者看起来相似，但关键区别在于：Rust 版本的函数签名（`-> Result<Output, Error>`）明确宣告了"我会失败"，而 JS 版本的签名（`async function process()`）完全隐藏了失败可能。

## 选型建议

| 如果你在… | 推荐策略 | 注意 |
|-----------|---------|------|
| 编写系统级代码（驱动、内核模块） | Linux 信号 + errno | 信号处理器中的操作受限 |
| 编写 Web 服务（Node.js） | async/await + try-catch | 必须处理 unhandledrejection |
| 编写高可靠性代码 | Rust `Result` 模式 | 即使不在 Rust 中也应采纳"错误即值"思维 |
| 编写游戏脚本（Cocos） | try-catch + 回调 err 检查 | 资源加载的异步错误最容易遗漏 |
| 编写库/API | Rust `Result` / 错误优先回调 | 让调用者从签名就能看到错误可能 |
