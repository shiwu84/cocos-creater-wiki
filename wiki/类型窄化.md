# 类型窄化

> [!abstract] 摘要
> 类型窄化（Narrowing）是 TypeScript 最独特的能力之一——编译器通过**分析运行时的控制流**（`if`/`switch`/`typeof`/`instanceof`/真值检查等），自动将变量的类型从宽泛的联合类型**收窄**为更精确的具体类型。它使得 TypeScript 的类型系统不必脱离 JavaScript 的惯用写法——你写正常的 JS 分支逻辑，TS 跟在后面自动推导出更精确的类型。这正是 TypeScript 的核心理念：**类型系统适应代码，而非代码适应类型系统**。

## 根本问题

TypeScript 的类型是**编译期**概念——变量声明为 `string | number`，编译器只知道它可能是这两种类型之一。但 JavaScript 的代码在**运行时**通过 `if`、`switch`、`typeof` 等控制流构造做出实际决策——在某一个分支里，代码逻辑本身就隐含了"此时它一定是 `string`"这一事实。

类型窄化要解决的问题就是**这座桥梁**：让编译器跟随你的控制流逻辑，在特定代码路径中自动推导出更精确的类型。你不需要手动类型断言，不需要 `as`——写出正常的运行时检查，TS 就懂了。

```typescript
function padLeft(padding: number | string, input: string): string {
  if (typeof padding === "number") {
    // 此处 padding 的类型被窄化为 number
    return " ".repeat(padding) + input;
  }
  // 经过 if 的排除后，此处 padding 的类型被窄化为 string
  return padding + input;
}
```

TypeScript 在此例中识别到 `typeof padding === "number"` 是一个**类型守卫（type guard）**——一条运行时检查，其结果能让编译器在某个作用域中进一步确定变量的实际类型。

## 窄化机制详解

### 1. `typeof` 类型守卫

JavaScript 的 `typeof` 操作符返回一组固定的字符串：`"string"`、`"number"`、`"bigint"`、`"boolean"`、`"symbol"`、`"undefined"`、`"object"`、`"function"`。TypeScript 对 `typeof` 有特殊的编译器知识——它知道 `typeof` 在不同 JS 值上的语义，并能利用它来窄化类型。

**重要陷阱**：`typeof null === "object"` 是 JavaScript 的历史遗留 bug。TypeScript 知道这一点，所以 `typeof x === "object"` 会把 `null` 仍然保留在窄化后的类型中：

```typescript
function printAll(strs: string | string[] | null) {
  if (typeof strs === "object") {
    // strs 窄化为 string[] | null —— null 没有被排除！
    for (const s of strs) {
      // 错误：strs 可能为 null
    }
  }
}
```

因此，先检查真值再检查 `typeof` 是常见模式：

```typescript
function printAll(strs: string | string[] | null) {
  if (strs && typeof strs === "object") {
    // strs 窄化为 string[] —— null 被真值检查排除了
    for (const s of strs) {
      console.log(s); // OK
    }
  }
}
```

### 2. 真值窄化（Truthiness Narrowing）

JavaScript 在条件上下文中会将值强制转为布尔值：`0`、`NaN`、`""`、`0n`、`null`、`undefined` 为 falsy，其余为 truthy。TypeScript 在 `if`、`&&`、`||`、`!` 等布尔上下文中利用真值信息窄化类型：

```typescript
function multiplyAll(
  values: number[] | undefined,
  factor: number
): number[] | undefined {
  if (!values) {
    // values 窄化为 undefined
    return values;
  }
  // values 窄化为 number[]
  return values.map((x) => x * factor);
}
```

**易错陷阱**：真值窄化对空字符串不敏感。`if (strs)` 对 `""` 也是 falsy，如果函数需要处理空字符串，只用真值检查可能遗漏：

```typescript
// 不推荐：空字符串被跳过
function printAll(strs: string | string[] | null) {
  if (strs) {
    // strs === "" 走不到这里！
  }
}

// 推荐：精确检查 null
function printAll(strs: string | string[] | null) {
  if (strs !== null) {
    // 现在空字符串也能正常处理
  }
}
```

### 3. 等值窄化（Equality Narrowing）

TypeScript 利用 `===`、`!==`、`==`、`!=` 进行窄化。这里有两个层面的智能：

**（a）共同类型推导**：当两个联合类型变量被比较为相等时，TS 推导出它们必须是两者共享的类型：

```typescript
function example(x: string | number, y: string | boolean) {
  if (x === y) {
    // x 和 y 都窄化为 string —— 这是两者唯一的共同类型
    x.toUpperCase(); // OK
    y.toLowerCase(); // OK
  } else {
    // x 恢复为 string | number，y 恢复为 string | boolean
  }
}
```

**（b）`null`/`undefined` 特殊处理**：JS 的 `== null` 同时匹配 `null` 和 `undefined`（反之亦然），TypeScript 完全理解这个语义：

```typescript
interface Container {
  value: number | null | undefined;
}

function multiplyValue(container: Container, factor: number) {
  if (container.value != null) {
    // container.value 窄化为 number —— null 和 undefined 都被排除了
    container.value *= factor; // OK
  }
}
```

### 4. `in` 操作符窄化

JavaScript 的 `in` 操作符检查对象或其原型链上是否存在某个属性。TypeScript 将其作为类型守卫——"有某属性的类型"和"没有某属性的类型"在 `in` 的两个分支中分别窄化：

```typescript
type Fish = { swim: () => void };
type Bird = { fly: () => void };

function move(animal: Fish | Bird) {
  if ("swim" in animal) {
    animal.swim(); // 窄化为 Fish
  } else {
    animal.fly();  // 窄化为 Bird
  }
}
```

**可选属性注意**：如果类型包含了可选属性，`in` 检查中该类型会同时出现在 true 和 false 两个分支——因为可选属性在运行时可能存在也可能不存在：

```typescript
type Human = { swim?: () => void; fly?: () => void };

function move(animal: Fish | Bird | Human) {
  if ("swim" in animal) {
    // animal: Fish | Human  —— Human 也有 swim（可选）
  } else {
    // animal: Bird | Human  —— Human 也可能没有 swim
  }
}
```

### 5. `instanceof` 窄化

JavaScript 的 `instanceof` 检查对象的原型链。TypeScript 将其识别为类型守卫，尤其适用于 class 构造的实例：

```typescript
function logValue(x: Date | string) {
  if (x instanceof Date) {
    console.log(x.toUTCString()); // 窄化为 Date
  } else {
    console.log(x.toUpperCase()); // 窄化为 string
  }
}
```

> `instanceof` 在跨 realm（如 iframe）的场景下可能失效——不同 realm 的构造函数是不同的对象。在纯 TypeScript/Node 环境中通常无需担忧。

### 6. 赋值窄化（Assignment Narrowing）

当给变量赋新值时，TypeScript 会参考右侧值的类型来更新左侧变量的**观察类型**（observed type）。但注意：赋值检查始终基于**声明类型（declared type）**，而非当前的观察类型：

```typescript
let x = Math.random() < 0.5 ? 10 : "hello world!";
//    ^? 声明类型为 string | number

x = 1;
//  ^? 观察类型为 number
x = "goodbye!";
// 此时 x 的观察类型更新为 string —— 依然在声明类型 string | number 范围内

x = true;  // 错误：boolean 不在声明类型 string | number 中
```

这意味着：即使某次赋值后 `x` 的观察类型是 `number`，你仍然可以后续赋值为 `string`——只要赋值的新值类型在声明类型的范围内。

### 7. 控制流分析（Control Flow Analysis）

上述所有机制的基础是 TypeScript 的**控制流分析**引擎。它不只是简单地扫描 `if` 中的类型守卫——它真正分析代码的**可达性（reachability）**，追踪变量在每一个代码位置上的精确类型。

当一个分支提前 `return` 或 `throw`，后续代码中的变量会被自动剔除已被排除的类型成员：

```typescript
function padLeft(padding: number | string, input: string) {
  if (typeof padding === "number") {
    return " ".repeat(padding) + input; // 此处 return，后续代码不可达
  }
  // 编译器知道：能走到这一步，padding 一定不是 number
  return padding + input; // padding 窄化为 string
}
```

控制流可以多次分裂和合并——每次赋值和类型守卫都会更新变量的类型状态：

```typescript
function example() {
  let x: string | number | boolean;

  x = Math.random() < 0.5;
  //  ^? boolean

  if (Math.random() < 0.5) {
    x = "hello";
    //  ^? string
  } else {
    x = 100;
    //  ^? number
  }
  // 控制流合并后，x 的类型重新变为 string | number
  return x;
}
```

> 这是 TypeScript 类型系统的基石能力——如果没有控制流分析，每一次类型窄化都需要开发者手动断言，TS 对 JS 代码的"零摩擦"体验将完全丧失。

## 进阶窄化

### 8. 类型谓词（Type Predicates）——自定义类型守卫

当内建的类型守卫不够用时，你可以通过**类型谓词**（`parameterName is Type`）定义自己的类型守卫函数。编译器会信任你的判断逻辑：

```typescript
type Fish = { swim: () => void };
type Bird = { fly: () => void };

function isFish(pet: Fish | Bird): pet is Fish {
  return (pet as Fish).swim !== undefined;
}

let pet = getSmallPet(); // Fish | Bird

if (isFish(pet)) {
  pet.swim(); // 窄化为 Fish
} else {
  pet.fly();  // 窄化为 Bird —— TS 知道 else 分支里一定不是 Fish
}
```

类型谓词的强大之处：
- **双向窄化**：`isFish` 返回 `true` 时窄化为 `Fish`，返回 `false` 时在 `else` 分支中排除 `Fish`
- **数组过滤**：`zoo.filter(isFish)` 的结果类型自动被推断为 `Fish[]`（而非原始的 `(Fish | Bird)[]`）
- **箭头函数谓词**：`zoo.filter((pet): pet is Fish => { ... })` 支持内联类型谓词

> [!warning] 类型谓词的信任边界
> 类型谓词是你对编译器的**承诺**——如果 `isFish` 的实现逻辑有误（比如误判为 true），编译器不会报错，但运行时会出现类型不匹配的错误。类型谓词本质上是**不安全的窄化**，需要你保证实现与声明的正确性。

### 9. 可辨识联合（Discriminated Unions）——最强大的窄化模式

可辨识联合是 TypeScript 中与窄化配合最紧密的设计模式，直接继承自函数式语言的代数数据类型（ADT）。其核心思想：

- 联合类型的每个成员都拥有一个**同名属性**，该属性的值是**字面量类型**
- 检查这个字面量属性的值，即可**无歧义地**确定当前属于联合类型的哪个成员
- 确定成员后，该成员独有的属性也随之可用

**反模式**（所有字段都设为可选）：

```typescript
interface Shape {
  kind: "circle" | "square";
  radius?: number;       // 问题：radius 对于 Square 也不应该存在
  sideLength?: number;   // 问题：sideLength 对于 Circle 也不应该存在
}

function getArea(shape: Shape) {
  if (shape.kind === "circle") {
    return Math.PI * shape.radius ** 2;
    //                   ~~~~~~ 错误：radius 可能为 undefined（可选属性）
  }
}
```

**正确模式**（每个成员独立定义，用 `kind` 区分）：

```typescript
interface Circle {
  kind: "circle";
  radius: number;        // 必选——Circle 一定有 radius
}

interface Square {
  kind: "square";
  sideLength: number;    // 必选——Square 一定有 sideLength
}

type Shape = Circle | Square;

function getArea(shape: Shape) {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;   // shape 窄化为 Circle
    case "square":
      return shape.sideLength ** 2;          // shape 窄化为 Square
  }
}
```

`kind` 属性在这里被称为**可辨识属性（discriminant）**。检查 `shape.kind` 的值（`"circle"` 或 `"square"`）后，TypeScript 自动：
1. 从 `Shape` 联合中排除不匹配的成员
2. 将 `shape` 窄化为该成员的精确类型
3. 该成员的所有必选属性立即可用——无需非空断言

> 可辨识联合的核心洞察：**用类型系统编码"一种数据有多种可能的形状"这一现实概念**。它比 OO 继承更直接——不需要基类、不需要虚方法、不需要运行时多态，只是在编译期通过区分标记来"知道当前是哪种形状"。这正是函数式编程中 sum type 的思想。

### 10. `never` 类型与穷尽性检查（Exhaustiveness Checking）

当窄化过程穷尽了所有可能，TypeScript 会将变量的类型标记为 `never`——一个**不包含任何值**的底层类型。`never` 的关键特性是：**never 可以赋值给任何类型，但没有任何类型可以赋值给 never（除了 never 本身）**。

利用这个特性，可以实现**穷尽性检查**——确保 switch 语句覆盖了联合类型的所有成员：

```typescript
function getArea(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.sideLength ** 2;
    default:
      // shape 在此处类型为 never，因为 Circle 和 Square 都已被处理
      const _exhaustiveCheck: never = shape;
      return _exhaustiveCheck;
  }
}
```

**穷尽性检查的工程价值**：当你日后向 `Shape` 联合添加新成员（如 `Triangle`）时，`default` 分支中的 `never` 赋值会立即报编译错误——因为新增的 `Triangle` 会使得 `shape` 在 default 分支中的类型变为 `Triangle`（而非 `never`），而 `Triangle` 无法赋值给 `never`。这相当于编译期的"你忘了处理新 case"提醒。

> 这个机制与 **Rust 的 match 穷尽性检查** 在精神上一模一样——都是利用类型系统保证"所有分支都已覆盖"，只是 Rust 在语言层面原生支持，TS 通过 `never` 技巧实现。

## 跨领域连接

### TypeScript ↔ Rust：可辨识联合与模式匹配

TS 的可辨识联合与 Rust 的 `enum` + `match` 在理念上高度共鸣：

| 概念 | TypeScript | Rust |
|------|-----------|------|
| **联合类型的定义** | `type Shape = Circle \| Square` | `enum Shape { Circle(f64), Square(f64) }` |
| **区分标记** | 字面量 `kind` 属性 | 枚举变体构造器 |
| **分支处理** | `switch (shape.kind)` + 窄化 | `match shape { ... }` |
| **穷尽性检查** | `never` 技巧 | 编译器原生强制 |

两者都在解决同一个根本问题：**如何在编译期安全地处理"数据有多种可能形状"的情况**。Rust 的解法更严格（编译器原生穷尽检查），TS 的解法更灵活（依靠结构类型和控制流分析，不强制枚举所有分支）。

### TypeScript → Cocos Creator：泛型窄化模式

Cocos Creator 中广泛使用泛型 + 窄化的组合模式，最典型的是 `getComponent<T>()`：

```typescript
const sprite = node.getComponent(Sprite);
// sprite 类型窄化为 Sprite | null —— getComponent 返回具体类型
if (sprite) {
  sprite.color = Color.RED; // 真值窄化排除 null
}
```

这种模式在引擎 API 中随处可见——泛型提供了类型参数的捕获能力，窄化确保了运行时获取的对象确实是正确的类型。

### TypeScript ↔ 软件工程：类型安全的状态模式

可辨识联合本质上是**在类型层面实现了 State 设计模式**。传统的 OO 状态模式通过接口 + 多态来实现状态切换；TS 的可辨识联合用类型 + 窄化来实现：

```typescript
// 状态用可辨识联合建模
type RequestState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string }
  | { status: "error"; message: string };

function render(state: RequestState) {
  switch (state.status) {
    case "idle":    return "等待中";
    case "loading": return "加载中...";
    case "success": return state.data;      // 窄化后 data 必然存在
    case "error":   return `错误: ${state.message}`;
    default:
      // never 穷尽检查——新增状态时编译报错
      const _: never = state;
      return _;
  }
}
```

每个状态拥有的数据只在对应分支中可用——编译器保证你不会在 `"idle"` 状态下访问 `state.data`。这比 OO 模式更轻量（无需定义类层次），也更安全（编译器自动跟踪状态变化）。

### TypeScript ← JavaScript：运行时操作符的静态投影

类型窄化之所以"感觉像普通 JS"，是因为 TS 的类型守卫直接建立在一组 JS 运行时操作符之上：

| JS 运行时操作符 | TS 类型守卫 | 窄化语义 |
|----------------|------------|---------|
| `typeof x` | `typeof x === "string"` 等比较 | 按 JS 的 typeof 语义窄化 |
| `x instanceof C` | `x instanceof Date` | 按原型链向下窄化 |
| `"key" in x` | `"key" in obj` | 按属性存在性窄化 |
| `x === y` / `x == y` | 等值比较 | 按共同类型窄化；`== null` 同时排除 `null` 和 `undefined` |
| `if (x)` | 真值检查 | 排除 falsy 值（含 `null`/`undefined`） |

TypeScript 不发明新的运行时语义——它只是"理解"JS 已有的运行时检查，并将这种理解反馈回类型系统。这是 TS 设计哲学的精髓：**做 JS 的类型层面投影，而非另起炉灶**。
