# 智能指针

> [!abstract] 摘要
> 智能指针（Smart Pointers）是行为像指针但携带额外元数据和能力的类型。在 Rust 的所有权系统中，智能指针解决了常规引用 `&T` 和 `&mut T` 无法覆盖的场景：堆分配（`Box<T>`）、多所有者的共享所有权（`Rc<T>`/`Arc<T>`）、在不可变引用下修改内部数据（`RefCell<T>`）。它们不是"绕过所有权规则"，而是在所有权框架内提供了更高阶的抽象。所有智能指针的共性来自两个关键 trait：`Deref`（像引用一样访问内部数据——deref 强制转换让智能指针在使用时透明如引用）和 `Drop`（所有者在离开作用域时自动清理资源——Rust 的 RAII 机制）。`Rc<RefCell<T>>` 组合是 Rust 在单线程下实现"共享可变状态"的模式——将借用检查从编译期推迟到运行时，保留了安全性但换取了灵活性（违反规则时 panic 而非编译错误）。

## 根本问题：常规引用的局限性

Rust 的常规引用 `&T` 和 `&mut T` 提供了编译期验证的安全性，零运行时开销，是所有场景的最佳默认选择。但有一些场景，常规引用无法胜任：

1. **编译期大小未知的类型**：递归类型（如链表、树节点）的大小在编译时无法确定，因为类型定义中包含自身——Rust 必须在编译期知道每个类型的确切大小。
2. **多所有者共享**：图结构、共享缓存等场景中，数据被多个部分同时拥有——但所有权的核心规则是"每个值只有一个所有者"。
3. **内部可变性**：在持有不可变引用的上下文中需要修改数据（如 mock 对象）——借用规则禁止通过 `&self` 修改内部状态。
4. **大量数据的低成本转移**：栈上拷贝大块数据代价高——需要只拷贝指针而数据留在原地。

常规引用是**零成本且最安全的**，但同时也是**最严格的**。智能指针在保持安全的前提下，为上述场景提供了结构化解决方案。

## 智能指针的共性基础：Deref 与 Drop

所有智能指针的强大能力来自两个核心 trait。

### Deref Trait：让智能指针透明如引用

`Deref` trait 允许重载解引用运算符 `*` 的行为。当你在一个类型上实现 `Deref`，就可以用 `*` 运算符访问其内部数据，就像它是常规引用一样：

```rust
pub trait Deref {
    type Target: ?Sized;
    fn deref(&self) -> &Self::Target;
}
```

`deref` 返回一个指向内部数据的引用。当编译器遇到 `*x` 时，实际上将其转换为 `*(x.deref())`。这样，任何实现了 `Deref` 的类型在使用 `*` 时都表现得像引用。

#### Deref 强制转换（Deref Coercion）

Deref 强制转换是比 `*` 运算符更重要的特性：当函数参数类型与传入的引用类型不匹配时，Rust 编译器会自动调用一系列 `deref` 方法进行类型转换。例如：

```rust
fn hello(name: &str) {
    println!("Hello, {name}!");
}

let m = MyBox::new(String::from("Rust"));
hello(&m);
// Deref 强制转换链: &MyBox<String> → deref → &String → deref → &str
```

编译器在编译期完成这些转换，**零运行时开销**。三种 Deref 强制转换规则：

| 来源 | 目标 | 条件 |
|------|------|------|
| `&T` | `&U` | `T: Deref<Target=U>` |
| `&mut T` | `&mut U` | `T: DerefMut<Target=U>` |
| `&mut T` | `&U` | `T: Deref<Target=U>`（可变引用可转为不可变引用，反过来不行） |

> [!important] DerefMut 的不对称性
> 可变引用可以强制转换为不可变引用（因为可变引用是数据的唯一引用，转为不可变是安全的），但**不可变引用永远不能转为可变引用**——借用规则无法保证当前不可变引用是唯一的。

### Drop Trait：RAII 的 Rust 实现

`Drop` trait 定义值离开作用域时的清理行为：

```rust
pub trait Drop {
    fn drop(&mut self);
}
```

当实现了 `Drop` 的值离开作用域时，编译器自动插入 `drop` 调用。**不能显式调用** `drop` 方法（编译错误），因为 Rust 在作用域结束时仍会自动调用一次——导致 double free。如需提前释放，使用 `std::mem::drop`（prelude 中的函数，接收所有权并立即丢弃）。

变量按创建顺序的**逆序**被丢弃——后创建的先释放。这保证了资源间依赖关系的正确清理。

> [!note] RAII（Resource Acquisition Is Initialization）
> Drop trait 是 Rust 实现 RAII 的机制。资源（内存、文件句柄、锁、socket）的获取绑定到值的创建，资源的释放绑定到值的析构。开发者不需要手动 `close`/`free`/`unlock`，编译器保证资源一定被释放。这也是 [[所有权与借用]] 中"值离开作用域自动释放"的技术基础。

## Box<T>：堆上的唯一所有权

`Box<T>` 是最简单的智能指针——将数据存储在堆上，在栈上保留一个固定大小的指针。它本身没有性能开销（零成本抽象），也没有太多额外能力。

### 三种核心使用场景

**1. 递归类型（编译期大小未知）**

递归类型在定义中包含自身，导致编译器无法计算其大小。`Box<T>` 提供了间接层——指针大小固定，打破递归链：

```rust
// 编译错误：类型有无限大小
// enum List { Cons(i32, List), Nil }

// 使用 Box 解决：指针大小固定
enum List {
    Cons(i32, Box<List>),
    Nil,
}
```

`Cons` 变体需要的空间 = `i32` 大小 + Box 指针大小。无论递归多深，每个节点的布局都是确定的。

**2. 大量数据的高效转移**

转移 `Box<T>` 的所有权时只拷贝栈上的指针（8 字节），堆上的数据保持不动。对比直接在栈上持有大结构体，转移所有权意味着复制整个结构体。

**3. Trait 对象（dyn Trait）**

`Box<dyn Trait>` 是 Rust 中动态分发的主要形式。`dyn Trait` 编译期大小未知，必须放在指针后面：

```rust
let components: Vec<Box<dyn Draw>> = vec![
    Box::new(Button { label: "OK" }),
    Box::new(TextField { placeholder: "name" }),
];
// 所有实现 Draw 的异构类型放进同一个集合
```

> [!warning] Box 不是引用计数的
> `Box<T>` 是**独占所有权**——同一时刻只有一个所有者。需要多所有者时使用 `Rc<T>` 或 `Arc<T>`。

## Rc<T>：单线程引用计数

`Rc<T>`（Reference Counted）允许多个部分**共享**同一份数据的所有权。它维护一个内部计数器（`strong_count`），记录当前有多少个 `Rc<T>` 指向这份数据：

```rust
use std::rc::Rc;

let a = Rc::new(List::Cons(5, Rc::new(List::Nil)));
// a 的 strong_count = 1

let b = List::Cons(3, Rc::clone(&a));
// strong_count = 2

let c = List::Cons(4, Rc::clone(&a));
// strong_count = 3
```

当某个 `Rc<T>` 离开作用域时，计数器减 1。计数器归零时，堆上的数据被释放。

### Rc::clone vs clone

`Rc::clone(&a)` **不进行深拷贝**——它只增加引用计数，是一个极低成本的操作。而 `a.clone()` 对于大多数类型意味着深拷贝。Rust 社区约定使用 `Rc::clone` 来在代码中显式区分"引用计数操作"和"数据深拷贝"。

### 限制：只能提供不可变访问

`Rc<T>` 实现 `Deref<Target=T>`，但只提供 `&T`——不可变引用。这是借用规则的体现：如果 `Rc<T>` 同时提供多个可变引用，就违反了"同一时刻只能有一个可变引用"的规则。需要修改共享数据时，必须使用 `RefCell<T>` 配合。

## RefCell<T>：内部可变性

`RefCell<T>` 是 Rust 中**内部可变性**（Interior Mutability）模式的实现。它允许你在持有**不可变引用**到 `RefCell<T>` 本身时，修改其**内部数据**。

### 根本机制：将借用检查从编译期移到运行时

常规引用的借用规则在编译期强制执行——违反则编译不过。`RefCell<T>` 将同样的规则推迟到**运行时**执行：

| 维度 | 常规引用 `&T` / `&mut T` | `RefCell<T>` |
|------|--------------------------|-------------|
| 检查时机 | 编译期 | 运行时 |
| 违反后果 | 编译错误 | **panic**（`BorrowError` / `BorrowMutError`） |
| 性能 | 零开销 | 运行时计数器维护开销 |
| 灵活性 | 编译器保守，可能拒绝正确程序 | 允许编译器无法静态验证的正确模式 |

运行时借用的 API：

```rust
// 获取不可变借用
let r: Ref<T> = ref_cell.borrow();         // 多个 Ref<T> 可共存

// 获取可变借用
let rm: RefMut<T> = ref_cell.borrow_mut(); // 同时只能有一个 RefMut<T>

// 违反规则时 panic：
// - borrow_mut() 时已有 Ref<T> 存在 → panic
// - borrow_mut() 时已有另一个 RefMut<T> 存在 → panic
```

`Ref<T>` 和 `RefMut<T>` 都实现了 `Deref`，使用起来像常规引用。它们离开作用域时自动减少内部的借用计数。

### 为什么需要内部可变性？

编译器的借用检查是**天生保守的**——对于任何它不能 100% 保证安全的模式，一律拒绝。但存在一些**客观上安全但编译器无法证明**的场景：

**典型场景：Mock 对象测试**

```rust
pub trait Messenger {
    fn send(&self, msg: &str);
}

pub struct LimitTracker<'a, T: Messenger> {
    messenger: &'a T,
    value: usize,
    max: usize,
}

// 实现 set_value 时，需要调用 self.messenger.send()——但 trait 定义的 send(&self)
// 编译器通过 trait 签名知道 send 接收 &self（不可变引用），
// 因此调用 send 不会修改 messenger——这满足借用规则

// 但 MockMessenger 需要在 send 中记录消息（修改内部状态）：
struct MockMessenger {
    sent_messages: RefCell<Vec<String>>,
}

impl Messenger for MockMessenger {
    fn send(&self, message: &str) {
        // self 是 &self（不可变引用），但通过 RefCell 可以修改 sent_messages
        self.sent_messages.borrow_mut().push(String::from(message));
    }
}
```

这是内部可变性的经典应用：trait 定义的接口使用 `&self`（遵循"不可变优先"的 Rust 惯例），但实现者需要在内部记录状态。`RefCell<T>` 在不改变 trait 签名的前提下实现了这一点。

### 为什么是安全的？

`RefCell<T>` 内部的 `unsafe` 代码绕过了编译器的静态分析，但封装在安全的 API 中。运行时检查保证：
- 同一时刻只有一个可变借用，或多个不可变借用
- 不存在悬垂引用
- 违反规则的代码会 panic 而不是产生未定义行为

> [!conflict] 编译期 vs 运行时检查的权衡
> 将检查推迟到运行时意味着：bug 可能在开发的更后期甚至生产环境才暴露（panic），而不是在编译时就发现。同时还有轻微的性能开销。因此，**首选常规引用**，只有在编译器无法验证而你知道逻辑正确时才使用 `RefCell<T>`。

## Rc<RefCell<T>>：共享可变状态

这是 Rust 中最重要的智能指针组合模式。单独的 `Rc<T>` 只能提供不可变访问，单独的 `RefCell<T>` 只有单一所有者。二者组合：

```
Rc<RefCell<T>>
```

实现了：**多个所有者** + **对同一数据的可变访问**。

```rust
use std::rc::Rc;
use std::cell::RefCell;

let value = Rc::new(RefCell::new(5));

let a = Rc::new(Cons(Rc::clone(&value), Rc::new(Nil)));
let b = Cons(Rc::new(RefCell::new(3)), Rc::clone(&a));
let c = Cons(Rc::new(RefCell::new(4)), Rc::clone(&a));

*value.borrow_mut() += 10;
// 现在 a、b、c 中引用的 value 都变成了 15
```

`Rc<RefCell<T>>` 在语义上等同于其他语言中的"共享可变状态"——数据被多处持有且可以被修改。但 Rust 做出了明确的区别：

- **编译器不保证合法性**——借用规则推迟到运行时
- **违反则 panic**——比 C++ 的未定义行为安全，但可能在生产环境崩溃
- **显式标注**——`Rc<RefCell<T>>` 的类型签名明确告诉读者"这里有共享可变状态"
- **单线程限定**——`Rc<T>` 和 `RefCell<T>` 都未实现 `Send`，不能跨线程传递

> [!important] `Rc<RefCell<T>>` 是 Rust 版本的"共享可变状态"
> 在其他语言（JS、Python、Java）中，对象的字段总是可以通过任何引用修改——共享可变是**默认且不可见的**。在 Rust 中，共享可变必须**显式声明和操作**（`Rc<RefCell<T>>` + `borrow_mut()`），让潜在的危险模式在代码中醒目可见。

## Weak<T>：防止引用循环

引用循环是使用 `Rc<T>` 时的最大陷阱。当两个 `Rc<T>` 互相引用时，各自的 `strong_count` 永远无法归零，导致内存泄漏：

```rust
// 引用循环示例（简化）
// a → b → a 形成环，a 和 b 的 strong_count 各为 1
// 但没有任何外部变量持有它们 → 计数不为 0，永不释放
```

`Weak<T>` 是解决引用循环的工具。它是 `Rc<T>` 的**非拥有引用**——不增加 `strong_count`，不影响值的释放。通过 `Rc::downgrade` 创建，增加的是 `weak_count`：

| 类型 | 创建方式 | 影响计数 | 影响释放 |
|------|---------|---------|---------|
| `Rc<T>` | `Rc::clone` | `strong_count + 1` | 是——strong_count 为 0 才释放 |
| `Weak<T>` | `Rc::downgrade` | `weak_count + 1` | 否——只关注 strong_count |

使用 `Weak<T>` 指向的值时，必须调用 `upgrade()` 方法，返回 `Option<Rc<T>>`——如果原数据已被释放则返回 `None`。这保证了不会出现悬垂指针。

### 典型应用：树结构中子节点指向父节点

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;

struct Node {
    value: i32,
    parent: RefCell<Weak<Node>>,     // 弱引用——不拥有父节点
    children: RefCell<Vec<Rc<Node>>>, // 强引用——拥有子节点
}
```

- 父节点拥有子节点：`Rc<Node>`（强引用）——父节点存在则子节点不能被释放
- 子节点引用父节点：`Weak<Node>`（弱引用）——子节点被释放不影响父节点
- 双向引用但不形成循环：强引用从父到子，弱引用从子到父

当父节点离开作用域时，其 `strong_count` 归零，数据被释放。子节点通过 `upgrade()` 检测到父节点已不存在（返回 `None`），不会产生悬垂指针。

## Arc<T>：线程安全的引用计数

`Arc<T>`（Atomic Reference Counted）是 `Rc<T>` 的线程安全版本。区别在于引用计数的操作：

- `Rc<T>`：普通整数增减，非原子操作（快，但不线程安全）
- `Arc<T>`：原子操作（`AtomicUsize`），使用 CPU 的原子指令确保多线程安全（有轻微性能开销）

`Arc<T>` 的 API 与 `Rc<T>` 几乎一致——`Arc::clone` 增加计数，离开作用域时自动减计数。常与 `Mutex<T>` 组合使用：

```rust
use std::sync::{Arc, Mutex};

let counter = Arc::new(Mutex::new(0));
let counter_clone = Arc::clone(&counter);

thread::spawn(move || {
    let mut num = counter_clone.lock().unwrap();
    *num += 1;
}).join().unwrap();
```

`Arc<Mutex<T>>` 是 `Rc<RefCell<T>>` 的多线程等价物，详细讨论见 [[Rust 并发模型]]。

| 组合 | 线程 | 共享所有权 | 内部可变性 |
|------|------|----------|-----------|
| `Rc<RefCell<T>>` | 单线程 | `Rc` | `RefCell`（运行时借用检查，panic） |
| `Arc<Mutex<T>>` | 多线程 | `Arc`（原子计数） | `Mutex`（运行时锁，阻塞） |

## 智能指针选择指南

| 需求 | 使用的智能指针 |
|------|-------------|
| 堆分配，独占所有权 | `Box<T>` |
| 递归类型（编译期大小未知） | `Box<T>` |
| Trait 对象（动态分发） | `Box<dyn Trait>` |
| 单线程共享不可变数据 | `Rc<T>` |
| 单线程共享可变数据 | `Rc<RefCell<T>>` |
| 多线程共享不可变数据 | `Arc<T>` |
| 多线程共享可变数据 | `Arc<Mutex<T>>` |
| 在 `&self` 方法中修改内部状态 | `RefCell<T>`（或 `Cell<T>` 用于 Copy 类型） |
| 防止引用循环 | `Weak<T>`（配合 `Rc<T>`） |
| 大型数据的所有权转移 | `Box<T>`（只转移指针） |

> [!tip] 选择原则
> 1. 优先使用常规引用 `&T` 和 `&mut T`——它们是零成本且最安全的
> 2. 需要堆分配或用 `dyn Trait` 时用 `Box<T>`
> 3. 需要多所有者时先用 `Rc<T>`；需要修改内部数据再加 `RefCell<T>`
> 4. 多线程场景用 `Arc<T>` 和 `Mutex<T>`；单线程不要用它们（无必要的原子操作开销）

## 跨领域连接

### ↔ Linux：内存管理概念在 Rust 中的体现

- **`Box<T>` ≈ `malloc` + `free` 的编译期安全封装**：C 中调用 `malloc` 分配堆内存并用指针访问，必须记得调用 `free`。Rust 的 `Box<T>` 在编译期内嵌了 `malloc`，在 `Drop` 中内嵌了 `free`——开发者不会忘记释放，编译器也不会让释放过早或重复发生。这与 [[Linux 进程模型]] 中进程退出时内核回收所有资源的思想一致——值离开作用域时 Rust 自动清理。
- **`Rc<T>` ≈ 共享内存区域的引用计数管理**：Linux 中多个进程可以通过 `mmap` 共享同一块内存区域，内核维护引用计数决定何时真正回收。Rust 的 `Rc<T>` 在用户态做了同样的事——只不过作用域缩小到单线程内的多个数据结构。
- **`RefCell<T>` ≈ 运行时权限检查**：Linux 中进程访问内存页时，MMU 在**运行时**检查页表权限（rwx）。如果进程试图写只读页，MMU 触发 Page Fault。`RefCell<T>` 的运行时借用检查类似——在访问的**那一刻**检查是否有权限冲突，违反则 panic（而不是编译错误）。这与常规引用在**编译期**验证形成对比：常规引用像在程序装载时静态分析所有内存访问，`RefCell<T>` 像在每次内存访问时动态检查。

### ↔ JavaScript：引用计数与可变性哲学

- **`Rc<T>` ≈ JavaScript 引擎的引用计数 GC**：现代 JS 引擎（V8、SpiderMonkey）使用标记-清除为主、引用计数为辅的 GC 策略。`Rc<T>` 就是**纯引用计数**——没有标记-清除、没有分代、没有 GC 的全局暂停。优点是完全确定性的释放时机，缺点是无法处理循环引用（需要 `Weak<T>` 辅助）。JS 中的对象总是通过引用共享，且默认可变——Rust 将这两者拆分开来：是否共享由 `Rc<T>` 控制，是否可变由 `RefCell<T>` 控制。
- **`RefCell<T>` ≈ JS 对象的默认可变性**：在 JavaScript 中，任何持有对象引用的代码都可以修改对象的属性——所有对象天然具备"内部可变性"。Rust 的 `RefCell<T>` 将这种能力显式标注出来，让你清楚地看到哪里存在共享可变状态——JS 中这种模式是隐式且普遍的。
- **单线程事件循环与 `Rc<RefCell<T>>` 的天然契合**：JS 的单线程事件循环意味着不需要担心真正的并行数据竞争——但 JS 仍然有异步交错执行带来的竞态条件（Race Condition）。`Rc<RefCell<T>>` 的运行时借用检查与 JS 的单线程模型在安全性上等价——都是在单一执行线程上，不用担心数据竞争，但仍需注意逻辑层面的竞态条件。

### ↔ C++：智能指针的参照系

- **`Box<T>` ≈ `std::unique_ptr<T>`**：独占所有权的堆分配指针，不可复制、可移动。区别在于 Rust 的 `Box<T>` 没有空状态（不能为 null），而 C++ 的 `unique_ptr` 可以为 `nullptr`。
- **`Rc<T>` ≈ `std::shared_ptr<T>`**：非原子的引用计数共享指针。区别在于 `Rc<T>` 是不可变的（需配合 `RefCell<T>` 才能修改），而 `shared_ptr` 提供了 `operator*` 和 `operator->`。
- **`Arc<T>` ≈ 线程安全的 `std::shared_ptr<T>`**：原子引用计数。
- **`RefCell<T>` 没有直接的 C++ 等价物**：因为 C++ 的 `const` 不是强制的——你可以通过 `const_cast` 绕过，也可以在标记为 `const` 的方法中修改 `mutable` 成员。C++ 没有编译期借用检查，因此不需要 `RefCell<T>` 这种"借用检查运行时版本"——本质上 C++ 的默认可变性已经覆盖了 `RefCell<T>` 的能力范围，但没有 Rust 的安全保障。

### → 软件工程

- **RAII 是智能指针的根本设计模式**：`Box<T>` 的 `Drop` 实现自动释放堆内存，`Rc<T>` 的 `Drop` 自动减少引用计数并在归零时释放数据，`Arc<T>` 同样。这体现了 [[软件工程概述|软件工程]] 中的 **RAII（Resource Acquisition Is Initialization）** 原则——资源的生命周期绑定到对象的生命周期。这是 Rust 中贯穿所有智能指针的核心设计模式。
- **内部可变性作为观察者/回调模式**：`RefCell<T>` 解决的核心问题是——当一个对象需要在其方法中修改自身状态，但方法的签名是 `&self`（不可变引用）时怎么办？这在设计模式的语境中就是**观察者模式**和**回调模式**——被观察对象在通知观察者时（`&self`），观察者可能需要修改自身状态。`RefCell<T>` 是 Rust 对这一模式的解决方案。详见 [[脚本系统]] 中 Cocos Creator 的回调机制对比。
- **显式化共享可变状态**：`Rc<RefCell<T>>` 的类型签名本身就是一种文档——它告诉代码阅读者"这里有共享的可变状态，要小心"。这是 [[软件工程概述|软件工程]] 中"让不可见的复杂变得可见"的范例。
