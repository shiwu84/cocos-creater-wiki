# 设计模式

> [!abstract] 摘要
> 设计模式是对反复出现的软件设计问题的命名解决方案——它不只是代码模板，更是程序员之间共享的设计词汇。模式的价值在于**命名**：当你说"这里用观察者模式"，团队能迅速建立共同理解，而不需要解释为什么要有个列表、回调怎么注册。但模式不是银弹：错误地套用模式会导致过度工程——解决不存在的问题，增加不需要的复杂性。每个模式在特定语言范式下的形态各不相同——观察者在 JS 中是事件回调，在 Cocos 中是组件事件系统，在 Rust 中受到借用检查的约束。

## 根本问题：为什么需要模式

写完许多项目后你会发现：看似不同的业务问题，在代码结构层面其实是同一类问题。比如"用户点击按钮 → 更新 UI"和"服务器收到消息 → 推送通知"——它们都是"一个对象状态变化时通知一批依赖对象"。设计模式提取了这些可复用的**结构关系**，并给它们命名。

模式解决的三个根本问题：
1. **认知减负**：不需要每次都从零设计，可以用已知模式快速匹配
2. **沟通效率**：模式名承载了大量设计意图的信息量
3. **质量保证**：被广泛验证的模式自带稳健性——已知的边界条件和陷阱

## 模式的跨语言形态

同一个模式在不同语言中因范式差异而表现出不同的实现形式。理解这种"同模异形"比背诵 GoF 23 模式更重要。

### 观察者模式

| 语言/环境 | 实现形态 |
|-----------|---------|
| Cocos Creator | `node.on('event', callback)`、`EventTarget` 装饰器——引擎内建支持 |
| JavaScript | `addEventListener`、`EventEmitter`、手动回调列表——生态拼装 |
| Rust | 闭包 + `Vec<Box<dyn Fn(&Event)>>`——注意借用约束：观察者需要 `&self` 存取，可能被借用检查器阻止 |

Cocos 引擎将观察者模式作为一个底层机制深度集成：组件脚本通过 `@property` 暴露参数，引擎通过观察者模式监听属性变化。这不只是设计模式的应用——它是引擎架构的基本语言。

### 策略模式

| 语言/环境 | 实现形态 |
|-----------|---------|
| Rust | `trait` 天然是策略模式的类型安全表达——`trait SortStrategy` 的不同实现就是不同策略 |
| TypeScript | 函数类型 `(input: T) => Output` 作为策略——简洁但缺乏"策略集合"的语义约束 |
| JavaScript | 回调函数——最简洁也最松散，没有任何类型约束 |

Rust 的 trait 系统使策略模式成为语言内建机制，而非设计模式书上需要手动实现的结构。

### 面向对象模式在 Rust 中的特殊性

Rust 没有继承。GoF 的很多模式假设了继承体系（如模板方法模式的 `abstract class`→子类重写）。在 Rust 中：
- **模板方法** → 闭包参数化行为，`fn execute<F: FnOnce()>(f: F)` 
- **状态模式** → 类型驱动状态：将状态编码为不同类型，状态转换 = 类型变化；或 trait object 封装可变状态
- **装饰器** → 函数组合 + trait 约束

Rust TRPL 第 18 章专门讨论了状态模式的重写——从"用 trait object 模拟面向对象的可变对象状态"到"将状态编码为 Rust 的类型系统"。第二种方案消除了运行时检查（状态转换错误在编译时捕获），代价是状态转换需要重新赋值。

## 实践原则

### "模式即毒药"——何时不用模式

1. **问题还没出现**：不要为了"未来可能需要"而添加模式。YAGNI。
2. **模式带来复杂性 > 解决的问题**：观察者模式引入间接性——如果你只有 2 个回调，直接调用即可
3. **语言已有内建解决方案**：Rust 中不需要"单例模式"——`static` 和 `lazy_static` 已解决；JS 中不需要"迭代器模式"——ES6 迭代器协议内建

### 识别模式

与其背诵模式清单，更好的方法是：**当发现自己在解决一个"这感觉我在哪里见过"的问题时，停下来，想一下这个问题的结构**——这就是模式的真正用途。

## 注意事项

- **模式是描述性，不是规范性的**：如果代码不能自然地适应某模式，不要强行
- **模式的边界条件比模式本身更重要**：知道什么情况下观察者模式会变慢（大量订阅者）、什么情况下策略模式会爆炸（策略数量失控）比知道它的定义更重要
- **跨语言时重新评估模式**：Rust 的所有权模型让很多"经典"模式需要重新思考（如状态模式中可变借用的困境）
