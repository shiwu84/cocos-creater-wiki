# 结构化类型与类型兼容性

> [!abstract] 摘要
> TypeScript 的类型系统基于**结构化子类型（structural subtyping）**，而非主流静态语言（Java、C#）采用的**名义类型（nominal typing）**。核心规则只有一句话：**如果两个类型拥有相同的结构（成员），它们就兼容——不需要有相同的名称、不需要显式声明 `implements`、不需要继承关系**。这听起来很激进，但它是 TS 对 JavaScript 生态的最深层适应——JS 代码里到处是匿名对象字面量和函数表达式，开发者几乎从不写类继承却大量使用"形状匹配"。结构化类型让 TS 能自然地描述这些 JS 惯用模式，但也在一些边界上制造了与名义类型直觉完全不符的"惊喜"。

## 根本问题：类型兼容性应该看什么？

在名义类型系统（Java、C#）中，两个类型是否兼容取决于它们在源代码中**声明的名字和继承关系**：

```java
// Java（名义类型）——以下代码是错误
interface Pet { String getName(); }
class Dog { public String getName() { return "Rex"; } }
Pet pet = new Dog(); // 编译错误：Dog 没有 implements Pet
```

Java 要求 `Dog` **显式声明** `implements Pet` 才能赋值。即使 `Dog` 有完全相同的方法签名，编译器也不认——因为它看的是"名字"而非"形状"。

TypeScript 选择了完全相反的思路：

```typescript
interface Pet {
  name: string;
}

class Dog {
  name: string;
}

let pet: Pet = new Dog(); // OK——Dog 有 name: string，形状匹配
```

TS 并不检查 `Dog` 是否有 `extends Pet` 或 `implements Pet`，它只检查 `Dog` 的实例是否拥有 `Pet` 所要求的全部成员。这个设计决策贯穿 TypeScript 的整个类型兼容系统。

### 为什么 TypeScript 选结构类型？

TypeScript 的类型系统设计被以下现实驱动：

1. **JS 生态中匿名对象字面量泛滥**——配置对象、函数选项、API 响应等几乎从来不是类的实例，而是赤裸裸的 `{ }`。结构类型让这些字面量可以直接匹配接口。
2. **大量 JS 库以"鸭子类型"方式运作**——如 `Array#forEach` 接受任何有 `length` 和索引访问的对象（类数组）。结构类型让 TS 能精确描述这种模式。
3. **JS 开发者不习惯写显式声明**——强迫每个类型都声明 `implements` 会与 JS 社区的文化激烈冲突。

## 对象类型的兼容

结构类型的基本规则是：**`x` 与 `y` 兼容，当且仅当 `y` 至少拥有 `x` 中列出的所有成员，且每个成员的类型兼容**。`y` 可以拥有额外的属性，这不会影响兼容性。

```typescript
interface Pet {
  name: string;
}

let pet: Pet;
let dog = { name: "Lassie", owner: "Rudd Weatherwax" };
// dog 的推导类型是 { name: string; owner: string; }

pet = dog; // OK——dog 有 name: string，多出来的 owner 被忽略
```

赋值的检查方向是"目标 → 源"：编译器检查目标类型（`Pet`）的每个属性，在源类型（`dog` 的类型）中是否存在兼容的对应属性。注意检查只关注目标的成员——源的额外成员不影响兼容性。

这个规则同样适用于函数参数：

```typescript
function greet(pet: Pet) {
  console.log("Hello, " + pet.name);
}

greet(dog); // OK——dog 的形状满足 Pet 的要求
```

### `readonly` 不影响兼容性

类型兼容检查中，`readonly` 修饰符**不参与比较**：

```typescript
interface Person {
  name: string;
  age: number;
}

interface ReadonlyPerson {
  readonly name: string;
  readonly age: number;
}

let writablePerson: Person = { name: "Alice", age: 30 };
let readonlyPerson: ReadonlyPerson = writablePerson; // OK

// 但注意：通过 writablePerson 修改后，readonlyPerson 的值也会变
writablePerson.age++;
console.log(readonlyPerson.age); // 31——readonly 只是编译期约束，运行时无保护
```

TypeScript 认为 `readonly` 是一种**使用意图**，而非类型结构的一部分——即便目标属性是 readonly，源是可写的也无妨，因为那只意味着源的使用者可能意外修改（通过别名效应），但赋值本身是安全的。

### 可选属性与兼容性

可选属性在兼容检查中的行为与必选属性一致——目标是可选属性，源有或没有该属性都可以：

```typescript
interface Config {
  color?: string;
  width?: number;
}

// 少提供可选属性——OK
let a: Config = { color: "red" };
let b: Config = { width: 100 };
let c: Config = {};  // OK，两个可选属性都没提供

// 多提供额外属性——在结构类型中本该 OK，但...见下文
```

### 过剩属性检查——结构类型规则的例外

这是结构化类型系统中最令人困惑的"例外"：对象字面量在**直接赋值**或**作为参数直接传入**时，会触发额外的**过剩属性检查（excess property checking）**：

```typescript
interface SquareConfig {
  color?: string;
  width?: number;
}

function createSquare(config: SquareConfig) { /* ... */ }

// 错误：对象字面量触发过剩属性检查
createSquare({ colour: "red", width: 100 });
//            ~~~~~~ 'colour' does not exist in type 'SquareConfig'

// 绕过方式 1：先赋给变量——变量赋值不触发过剩属性检查
let squareOptions = { colour: "red", width: 100 };
createSquare(squareOptions); // OK

// 绕过方式 2：类型断言
createSquare({ colour: "red", width: 100 } as SquareConfig);
```

**为什么存在这个"例外"？** TypeScript 判断——对象字面量是"新鲜"的，开发者直接把字面量传给已知类型时，多余属性极大概率是拼写错误（如 `colour` vs `color`）。而通过中间变量传递时，多余的属性可能是故意保留的（中间变量可能用于多个用途）。这不是类型系统规则的不一致——过剩属性检查是**额外的、可选的严格化**，它只在对象字面量的直接上下文中生效，不改写结构类型的底层规则。

> 过剩属性检查是 TS 中最常见的"类型兼容性困惑"来源——很多人以为 `createSquare({ colour: "red" })` 报错意味着类型不兼容，但 `const x = { colour: "red" }; createSquare(x)` 却能通过。**这不是 bug——这是 TS 对 typo 保护 vs 结构灵活性的平衡设计**。

## 函数类型的兼容

函数兼容判断是结构类型系统中最复杂的部分，因为函数有自己的"形状"——参数列表 + 返回类型。

### 参数兼容：允许丢弃多余参数

在参数兼容性上，TypeScript 允许多余参数被"丢弃"——源函数可以比目标函数**少**声明参数：

```typescript
let x = (a: number) => 0;
let y = (b: number, s: string) => 0;

y = x; // OK——x 少一个参数，多余的参数被忽略
x = y; // 错误——y 有多余的必选参数 s，x 没有
```

这看似违反直觉，但它是为了适配 JavaScript 中极其常见的模式——回调函数经常只使用部分参数：

```typescript
let items = [1, 2, 3];

// forEach 的回调会收到 (item, index, array) 三个参数，
// 但只用第一个是完全合法且极其常见的
items.forEach((item) => console.log(item));
```

如果不允许丢弃参数，每个 `forEach` 回调都必须声明三个参数——这会严重破坏 TypeScript 的易用性。

### 返回类型：协变（Covariance）

函数返回类型的兼容是**协变的**——源函数的返回类型必须是目标函数返回类型的**子类型**（即至少包含目标所需的所有属性）：

```typescript
let x = () => ({ name: "Alice" });
let y = () => ({ name: "Alice", location: "Seattle" });

x = y; // OK——y() 返回的超集包含了 x() 所需的 name
y = x; // 错误——x() 缺少 location 属性
```

返回类型协变与对象类型兼容的规则完全一致——"有更多东西"的值可以赋值给"要更少东西"的类型。`x` 只需要 `name`，`y()` 返回了 `name + location`，足够满足。

### 参数双变（Bivariance）：有意的不安全性

这是 TypeScript 中最著名的"不健全（unsound）"设计——函数参数的兼容是**双变的**：源参数可以赋值给目标参数，**反之也行**：

```typescript
enum EventType { Mouse, Keyboard }

interface Event { timestamp: number; }
interface MyMouseEvent extends Event { x: number; y: number; }
interface MyKeyEvent extends Event { keyCode: number; }

function listenEvent(eventType: EventType, handler: (n: Event) => void) {
  /* ... */
}

// 不健全——但实用且常见
listenEvent(EventType.Mouse, (e: MyMouseEvent) => console.log(e.x + "," + e.y));
```

从类型安全角度看，这是有问题的——`listenEvent` 声明的 handler 参数是 `Event`，但调用方传了一个只接受 `MyMouseEvent` 的函数。如果 TypeScript 严格检查，这应该报错。

**但实际体验表明**：这种错误在实际开发中极少发生，而禁止这种模式会让大量常见的 JS 事件监听代码被迫使用类型断言或多余的转换。TypeScript 团队有意选择了实用性优先。

通过 `strictFunctionTypes` 编译选项可以开启严格的函数参数逆变检查，让上述代码报错。

### 可选参数和剩余参数的兼容

可选参数与必选参数在函数兼容检查中**互换**——也就是说可选参数不构成赋值障碍：

```typescript
function invokeLater(args: any[], callback: (...args: any[]) => void) {
  /* ... 用 args 调用 callback ... */
}

// 不健全但实用——invokeLater "可能"提供任意数量的参数
invokeLater([1, 2], (x, y) => console.log(x + ", " + y));
```

剩余参数在兼容检查中被视为**无限个可选参数的序列**——同样不健全，但避免了把每个回调的参数都标记为可选。

### 重载函数的兼容

当函数有多个重载签名时，目标的**每个**重载都必须能在源中找到兼容的签名——这确保源函数在所有目标函数可用的调用方式中都可用。

## 类的兼容

类在 TypeScript 中的类型兼容只比较**实例成员**——静态成员和构造函数不参与兼容检查：

```typescript
class Animal {
  feet: number;
  constructor(name: string, numFeet: number) {}
}

class Size {
  feet: number;
  constructor(numFeet: number) {}  // 完全不同的构造函数
}

let a: Animal;
let s: Size;

a = s; // OK——实例成员相同（都有 feet: number）
s = a; // OK
```

`Animal` 和 `Size` 的构造函数签名完全不同（参数类型和数量都不同），但在兼容检查中完全被忽略——只有 `feet: number` 参与了比较。

### `private`/`protected` → 名义类型的"烙印"

这是 TypeScript 结构类型系统中**唯一**引入名义行为的地方——当且仅当类有 `private` 或 `protected` 成员时，兼容检查要求这些成员必须**源自同一个类**：

```typescript
class Animal {
  private name: string;
  constructor(theName: string) {
    this.name = theName;
  }
}

class Rhino extends Animal {
  constructor() {
    super("Rhino");
  }
}

class Employee {
  private name: string;
  constructor(theName: string) {
    this.name = theName;
  }
}

let animal = new Animal("Goat");
let rhino = new Rhino();
let employee = new Employee("Bob");

animal = rhino;    // OK——Rhino extends Animal，private name 来自同一个类 Animal
animal = employee; // 错误！Employee 的 private name 与 Animal 的 private name 不是"同一个"
```

即使 `Animal` 和 `Employee` 的形状完全一样（都是 `{ name: string }`），`private` 修饰符充当了"身份标记"——编译器追踪 private/protected 成员的**来源类**，只有来源一致的才兼容。这允许子类和父类之间正常赋值，但阻止了结构相同但毫不相关的类之间意外赋值。

> 这个机制可以被利用来实现"品牌类型"（branded types）——故意添加 `private` 成员使得结构相同但语义不同的类型无法互相赋值。

## 枚举的兼容

数值枚举和 `number` 之间是**双向兼容**的，但不同枚举类型之间**互不兼容**：

```typescript
enum Status { Ready, Waiting }
enum Color { Red, Blue, Green }

let status = Status.Ready;
status = 1;           // OK——number 可赋值给枚举
status = Color.Green; // 错误！不同枚举类型互不兼容
```

字符串枚举更严格——它不与 string 双向兼容：

```typescript
enum StringStatus { Active = "active", Inactive = "inactive" }
let s = StringStatus.Active;
s = "active"; // 错误！string 不能赋值给字符串枚举
```

## 泛型与兼容

泛型类型的兼容规则仍然遵循结构类型原则——**类型参数只在被实际用作成员类型时才影响兼容性**：

```typescript
interface Empty<T> {}
let x: Empty<number>;
let y: Empty<string>;

x = y; // OK！Empty<T> 没有任何成员使用 T，所以所有 Empty<任意> 结构完全相同
```

当类型参数出现在成员中时，结构类型规则照常生效：

```typescript
interface NotEmpty<T> {
  data: T;
}
let a: NotEmpty<number>;
let b: NotEmpty<string>;

a = b; // 错误！data 的类型不同（number vs string），结构不匹配
```

未指定类型参数的泛型类型——所有类型参数被替换为 `any` 后比较：

```typescript
let identity = function <T>(x: T): T { return x; };
let reverse = function <U>(y: U): U { return y; };

identity = reverse; // OK——(x: any) => any 与 (y: any) => any 结构相同
```

## 类型兼容性：Subtype vs Assignment

TypeScript 内部有两种兼容检查机制：

| 机制 | 规则 | 应用场景 |
|------|------|---------|
| **Subtype 兼容** | 纯结构类型规则 | 类型间的"子类型"关系 |
| **Assignment 兼容** | Subtype + `any` 方向 + enum↔number | 赋值、函数参数传递 |

**关键差异在于 `any`**：任何类型都可以赋值给 `any`，`any` 也可以赋值给任何类型（除了 `never`）。`unknown` 更安全——任何类型可以赋值给 `unknown`，但 `unknown` 只能赋值给 `any`：

| 从 → 到 | any | unknown | object | void | undefined | null | never |
|---------|-----|---------|--------|------|-----------|------|-------|
| any | - | ✓ | ✓ | ✓ | ✓ | ✓ | ✕ |
| unknown | ✓ | - | ✕ | ✕ | ✕ | ✕ | ✕ |
| never | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | - |

> `never` 是底层类型（bottom type）——可以赋值给一切；`unknown` 是顶层类型（top type）——一切都可以赋值给它。

## 跨领域连接

### ↔ Rust：结构 trait vs 结构类型

Rust 的 trait 是**结构化的**——一个类型不需要"声明实现某个 trait"，只需要实际提供了 trait 要求的方法体。从这个意义上，Rust trait 与 TS 的结构类型在理念上一致：

```rust
// 不声明 impl Greet for Dog，但如果 Dog 有 greet()，就可以做为泛型参数
fn speak<T: Greet>(t: T) { t.greet(); }
```

但与 TS 的关键区别是：Rust 的类型兼容本身仍然是**名义的**——两个 `struct` 即使字段完全一样也是不同类型。TS 的结构化是贯穿始终的（类、接口、匿名类型全都看形状），而 Rust 只是 trait 约束层面的"结构匹配"。

| 维度 | TypeScript | Rust |
|------|-----------|------|
| **类型等价** | 结构等价：有相同成员就相同 | 名义等价：同名 struct 才相同 |
| **接口/Trait** | 结构匹配：有接口需要的成员就兼容 | Trait 约束：必须有 `impl Trait for Type`，但实现可自动推导（derive） |
| **私有化保护** | `private` 引入名义行为（同一类来源） | 模块可见性控制（`pub`），不参与 trait 判断 |

### ↔ Go：结构类型的另一个实践者

Go 语言也采用结构类型来定义接口：

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

// 任何有 Read 方法的类型自动满足 Reader——无需显式声明
```

TypeScript 的结构类型直接受 Go 等语言的影响——两者都在解决同一个问题：**如何在不牺牲静态类型安全的前提下，保留动态语言"形状匹配"的灵活性**。不同之处：Go 只在接口层面是结构化的（struct 仍然是名义类型），TS 从里到外全都是结构类型。

### → JavaScript：鸭子类型的静态形式化

JavaScript 社区的"鸭子测试"（duck test）——"如果它走起来像鸭子，叫起来像鸭子，那它就是鸭子"——本质上是运行时基于形状的行为判断。TS 的结构化类型是这个原则的**编译期形式化**：

```javascript
// JS 运行时——运行到出错才知道
function greet(pet) {
    console.log("Hello, " + pet.name);  // pet 必须有 name，但只在运行时暴露问题
}
```

```typescript
// TS 编译期——不执行代码就知道
function greet(pet: { name: string }) {
    console.log("Hello, " + pet.name);  // 编译器保证 pet 有 name: string
}
```

TS 的结构化类型系统提供了 **"无需改变 JS 惯用写法，但获得编译期形状验证"** 的体验。这就是为什么 TypeScript 能大规模成功——它接受的代码看起来还是 JavaScript。

### → 软件工程：结构类型作为设计模式基础设施

结构化类型系统自然地推动了若干软件工程实践：

- **适配器模式零开销**：在名义类型系统中，适配器需要显式的包装类和委托；在 TS 中，只要创建一个形状匹配的对象字面量，它自动就是"适配器"。
- **依赖倒置更轻量**：接口由消费者定义——`function process(config: { url: string })` 定义了它需要的"形状接口"，任何提供者只要返回 `{ url: string }` 即可满足，无需事先知道接口名称。
- **渐进类型化**：结构兼容让 JS → TS 迁移可以从局部开始——不需要一次性建立完整的类型继承图，逐步补充类型注解即可。
