# 迭代器与闭包

> [!abstract] 摘要
> 闭包和迭代器是 Rust 从函数式编程借鉴的两大零成本抽象。闭包是能捕获环境的匿名函数——不同于普通函数，它可以"记住"定义时所在作用域的变量；迭代器是惰性元素序列——在调用消费方法前不做任何计算。两者联合使用构成了 Rust 标准库中数据转换的核心范式（`iter().filter().map().collect()`），且编译后的代码与手写循环无性能差异——这是 Rust"零成本抽象"承诺的典型体现。

## 根本问题：代码复用与抽象的表现力

传统的 `for` 循环存在两个问题：
1. **关注点混在一起**：遍历逻辑、过滤条件、转换操作写在一个循环体内，阅读者需要在脑中解耦
2. **复用困难**：循环体无法作为参数传递给其他函数

闭包和迭代器分别解决了这两个问题：
- **闭包**让行为（过滤条件、转换规则）成为可传递的值
- **迭代器**将"遍历序列"这个通用模式封装为标准 trait

## 闭包：捕获环境的匿名函数

### 三种捕获模式

闭包根据函数体对捕获变量的操作，自动选择捕获方式：

| 捕获方式 | 对应引用类型 | 行为 |
|---------|-------------|------|
| 不可变借用 | `&T` | 闭包体只读被捕获值 |
| 可变借用 | `&mut T` | 闭包体修改被捕获值 |
| 获取所有权 | `T` | 使用 `move` 关键字强制转移所有权 |

### Fn trait 分级

闭包对环境值的处理方式决定了它实现哪些 `Fn` trait：

| Trait | 约束 | 适用场景 |
|-------|------|---------|
| `FnOnce` | 可被调用一次（所有闭包都实现） | `unwrap_or_else` |
| `FnMut` | 可多次调用，可能修改环境 | `sort_by_key`、`for_each` |
| `Fn` | 可多次调用，不修改环境 | `map`、`filter`、并发场景 |

trait 实现是自动、渐进的：只读环境的闭包同时实现 `FnOnce`、`FnMut`、`Fn`；可变借用的闭包实现 `FnOnce` 和 `FnMut`；获取所有权的闭包只实现 `FnOnce`。

### 闭包与 JS 闭包的对比

- **共同点**：都捕获词法作用域中的变量
- **关键差异**：Rust 的捕获方式由编译器根据使用方式自动推断（不可变/可变/所有权），编译期确保内存安全；JS 闭包引用计数管理生命周期，不区分借用和所有权

## 迭代器：惰性序列处理

### Iterator trait

```rust
pub trait Iterator {
    type Item;
    fn next(&mut self) -> Option<Self::Item>;
}
```

只需实现 `next`，就能获得 70+ 个方法的默认实现。

### 消费适配器 vs 迭代器适配器

- **消费适配器**（consuming adaptors）：调用 `next` 消耗迭代器，如 `sum()`、`collect()`、`count()`、`for_each()`
- **迭代器适配器**（iterator adaptors）：返回新迭代器，**惰性**执行，如 `map()`、`filter()`、`take()`、`skip()`、`enumerate()`

关键原则：**迭代器适配器直到消费方法被调用前，不做任何计算**。链式调用 `iter().filter().map()` 创建的是"计算描述"，而非实际执行。

### 三种迭代方法

- `iter()`：返回不可变引用迭代器 `&T`
- `iter_mut()`：返回可变引用迭代器 `&mut T`
- `into_iter()`：返回所有权迭代器，消耗原集合

## 零成本抽象

`for` 循环和迭代器的性能测试结果是相同的。Rust 编译器会将 `iter().filter().map().collect()` 展开为与手写循环等价的汇编，包括循环展开、边界检查消除等优化。这和 C++ 的"零开销原则"一致：不使用则无开销，使用则无法手写得更快。

这与 JS 形成鲜明对比：JS 的 `Array.prototype.map()` 会在每次调用时创建新数组（多次分配），而 Rust 的惰性迭代器只在最终的 `collect()` 时分配一次。

## 注意事项

- **闭包类型是唯一的**：每个闭包实例都有编译器生成的匿名类型，无法直接作为函数指针传递（需用泛型或 trait object）
- **`move` 闭包与线程**：将闭包传递给 `thread::spawn` 时必须用 `move`，确保数据所有权转移进新线程
- **惰性陷阱**：忘记在链末调用消费适配器时，编译器会警告迭代器未被使用——整个链什么也没做
