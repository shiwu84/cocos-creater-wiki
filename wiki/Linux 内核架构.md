# Linux 内核架构

> [!abstract] 摘要
> Linux 内核是操作系统的核心——它是唯一能直接访问硬件、管理 CPU/内存/I/O 等物理资源、并强制安全隔离的软件层。内核架构围绕四个核心设计：(1) **特权级别分離**——内核在 Ring 0（内核空间）运行，拥有对硬件的完全访问权；用户程序在 Ring 3（用户空间）运行，只能通过系统调用（syscall）向内核请求服务，每一次调用都是一次上下文切换（用户态→内核态→用户态）；(2) **宏内核 + 模块化**——Linux 是宏内核（monolithic kernel），所有核心子系统（进程管理、内存管理、文件系统、网络栈、设备驱动）共享同一内核地址空间以获得极致性能，但通过可加载内核模块（LKM）机制避免了传统宏内核的"重新编译一次改驱动"的僵化——模块可以按需加载和卸载；(3) **系统调用表**——用户程序不直接执行硬件指令，而是通过 C 库包装器触发软件中断（x86 上为 `int 0x80` 或 `syscall` 指令），内核在 syscall 表中根据唯一编号查找到对应函数执行，完成后将结果和 CPU 控制权返回用户空间；(4) **内核内存不可换出**——内核的代码和数据页永远不会被 swap 到磁盘，因为换出内核会使系统无法处理页面错误，形成逻辑死锁。理解内核架构是理解一切上层抽象的第一性原理前提——Rust 的 `unsafe` 块本质是穿越内核边界的系统调用包装，Node.js 的 libuv 用 epoll/kqueue 将内核 I/O 多路复用包装为事件循环，Cocos 引擎的所有文件读写、网络通信、GPU 操作最终都穿过内核。

## 根本问题：操作系统为什么需要一个内核？

一台计算机的硬件资源（CPU、内存、硬盘、网卡、GPU）是所有正在运行的程序共享的。没有任何中央管理的话，会出现三类根本矛盾：

1. **资源争抢**：两个程序同时往硬盘同一扇区写入数据——谁的数据被保留？谁的数据被损坏？
2. **安全越界**：一个程序能否读取另一个程序的内存？能否截获另一个程序的网络包？能否控制摄像头？
3. **硬件复杂性**：一块 NVMe SSD 和一块 SATA HDD 的读写指令完全不同——每个程序都要自己处理这些差异吗？

内核的存在就是为了解决这三个矛盾。它作为**唯一的硬件特权管理者**，让所有应用程序通过统一的、受限的接口使用硬件。

## 三层架构：硬件 → 内核 → 用户空间

Linux 操作系统可以垂直切分为三个抽象层次：

```
┌─────────────────────────────────────────────────────┐
│                    用户空间 (User Space)               │
│   Shell / GUI / 守护进程 / 应用程序                     │
│   ┌───────────────────────────────────────────────┐  │
│   │  C 库 (glibc) — 系统调用的包装层                │  │
│   │  write(), read(), fork(), mmap() ...          │  │
│   └───────────────────┬───────────────────────────┘  │
│                       │ 系统调用 (syscall)             │
│   ════════════════════╪═══════════════════════════   │
│                       ▼                               │
│              内核空间 (Kernel Space)                   │
│   ┌─────────────────────────────────────────────┐    │
│   │  进程管理 │ 内存管理 │ 文件系统 │ 网络栈      │    │
│   │  设备驱动 │ 中断处理 │ 系统调用表              │    │
│   └─────────────────────────────────────────────┘    │
│                       │                               │
│                       ▼                               │
│              硬件层 (Hardware)                         │
│   CPU / RAM / 磁盘 / 网卡 / GPU / USB ...             │
└─────────────────────────────────────────────────────┘
```

> [!tip] 关键理解
> 内核是**唯一**能够直接访问硬件的层次。用户空间的任何程序——包括以 root 运行的——都不能绕过内核直接操作硬件。这个"唯一通道"的设计是操作系统安全性不可动摇的基石。

三个层次之间的交互是单向的：用户程序只能通过系统调用请求内核服务，内核执行完毕后返回结果。用户程序永远无法"看到"内核空间的内存，内核也无法直接访问已被换出的用户空间页面（必须先将它换入）。

## 特权级别：Ring 0 vs Ring 3

CPU 硬件本身就支持分级的特权模型。x86 架构定义了四个保护环（Protection Rings）：

| 环 | 特权级别 | 运行内容 | 能力 |
|----|---------|---------|------|
| **Ring 0** | 最高 | 内核 | 执行所有 CPU 指令，访问所有内存和硬件端口 |
| Ring 1 | — | (Linux 未使用) | — |
| Ring 2 | — | (Linux 未使用) | — |
| **Ring 3** | 最低 | 用户程序 | 只能执行非特权指令，只能访问自己的虚拟地址空间 |

Linux 只使用两个环——Ring 0 和 Ring 3，这简化了设计且不影响安全保证。中间环（Ring 1、2）原本为设备驱动设计，但在实践中几乎从未被操作系统使用。

> [!note] 特权级别 ≠ 用户权限
> Ring 3 中运行的程序仍然可以是 root 身份——root 是一个用户空间概念（UID 0）。一个 root 进程仍然必须通过系统调用才能访问硬件。特权级别分离的是**指令执行权限**，用户权限管理（rwx、SUID）分离的是**资源访问权限**——两者是不同维度的安全机制。

跨环切换的成本很高。从 Ring 3 切换到 Ring 0（系统调用）需要：
- 保存用户空间的寄存器状态
- 切换到内核栈
- 切换页表（或至少改变访问权限）
- 执行内核代码
- 恢复用户状态并返回

每一次系统调用都是一次昂贵的上下文切换——这就是为什么高性能 I/O 框架（如 io_uring）的核心思路是**批量提交系统调用**以减少切换次数。

## 系统调用：穿越特权边界的受控通道

### 什么是系统调用？

**系统调用（syscall）** 是用户程序与内核交互的**唯一合法接口**。它不是一个普通的函数调用——它是一次从 Ring 3 到 Ring 0 的 CPU 特权级别切换。

内核向外暴露了一组固定的系统调用，每个调用有唯一的编号。用户程序不能随意新增系统调用——内核的系统调用表在编译时就已确定。这个"固定菜单"的设计确保了用户空间永远不能要求内核做未定义的事。

### 系统调用的执行流程

以一个最简单的 `write(1, "hello", 5)` 为例（向 stdout 输出 5 字节）：

```
用户程序
  │
  │ 1. 调用 C 库的 write() 函数
  ▼
glibc (用户空间)
  │
  │ 2. 将参数放入寄存器 (fd=1, buf="hello", count=5)
  │ 3. 将系统调用编号放入 rax (x86-64 上 write=1)
  │
  │ 4. 执行 syscall 指令（x86-64）或 int 0x80（x86）
  ▼
══════════ 陷入内核 (Trap) ══════════
  │
  ▼
内核陷阱处理程序
  │
  │ 5. 保存用户空间寄存器到内核栈
  │ 6. 切换页表到内核地址空间
  │ 7. 查 syscall table[1] → sys_write()
  │ 8. 执行 sys_write()：验证 fd、权限检查、写入数据
  │ 9. 将返回值放入 rax
  │ 10. 恢复用户空间寄存器
  │
  │ 11. 执行 sysret 指令（或 iret）
  ▼
══════════ 返回用户空间 ══════════
  │
  ▼
glibc
  │
  │ 12. 检查返回值，设置 errno（如果有错误）
  │ 13. 返回写入字节数给用户程序
  ▼
用户程序
```

### 核心系统调用一览

| 调用 | 编号 (x86-64) | 类别 | 功能 |
|------|-------------|------|------|
| `read` | 0 | 文件 I/O | 从文件描述符读取数据 |
| `write` | 1 | 文件 I/O | 向文件描述符写入数据 |
| `open` | 2 | 文件 I/O | 打开文件，返回文件描述符 |
| `close` | 3 | 文件 I/O | 关闭文件描述符 |
| `mmap` | 9 | 内存管理 | 将文件或匿名内存映射到进程地址空间 |
| `fork` | 57 | 进程管理 | 创建当前进程的副本（子进程） |
| `execve` | 59 | 进程管理 | 用新程序替换当前进程 |
| `exit` | 60 | 进程管理 | 终止当前进程 |
| `socket` | 41 | 网络 | 创建网络通信端点 |
| `epoll_create` | 213 | 网络 | 创建 epoll 实例（I/O 多路复用） |

> [!tip] 用 strace 观看系统调用
> ```bash
> strace ls          # 查看 ls 命令发出的每一个系统调用
> strace -c ls       # 统计各系统调用次数和耗时
> strace -e open ls  # 只跟踪 open() 系统调用
> ```
> `strace` 是理解用户程序与内核交互的最直接工具——它展示了每一个跨越特权边界的瞬间。

### 为什么系统调用不能是普通函数调用？

如果 `write()` 只是一个普通的函数调用，会有什么问题？

1. **没有特权提升**：用户代码无法执行 I/O 指令（x86 的 `in`/`out` 指令在 Ring 3 下会触发保护异常）
2. **没有地址空间隔离**：用户代码可以直接读写内核内存，一个野指针就可能破坏整个系统的关键数据结构
3. **没有权限检查**：内核无法在执行操作前验证调用者是否有权限（文件权限、进程凭证等）

系统调用的"陷阱"机制正是为了强制切换到一个可信的上下文后再执行危险操作。

## 宏内核 vs 微内核：Linux 的设计选择

### 宏内核（Monolithic Kernel）

Linux 采用宏内核设计：**所有核心子系统都在同一个内核地址空间内运行**。进程管理器、内存管理器、文件系统、网络栈、设备驱动——全部共享同一个内核地址空间，可以互相直接调用。

**优势**：
- **性能极高**：子系统间通信是普通函数调用，零开销
- **简单统一**：不需要复杂的 IPC（进程间通信）机制来协调子系统

**代价**：
- **稳定性风险**：一个驱动的 bug 可能崩溃整个内核（因为所有模块共享内核地址空间，没有隔离）

### 微内核（Microkernel）

与之对比，微内核（如 Minix、QNX、seL4、macOS/iOS 的 XNU 混合内核）将大多数服务（文件系统、网络栈、驱动）移出内核，作为独立的用户空间进程运行。微内核本身只提供最小功能集：地址空间管理、线程调度、IPC。

```
宏内核                          微内核
┌─────────────┐                ┌─────────────┐
│ 进程管理     │                │  微内核核心   │
│ 内存管理     │                │ (IPC, 调度)  │
│ 文件系统     │                └──────┬──────┘
│ 网络栈       │              ┌────────┼────────┐
│ 设备驱动     │         ┌────┴───┐ ┌──┴───┐ ┌──┴───┐
└─────────────┘         │文件系统│ │网络栈│ │驱动  │
   共享地址空间           │ (进程) │ │(进程)│ │(进程)│
                        └────────┘ └──────┘ └──────┘
                          各自独立地址空间，通过 IPC 通信
```

| 维度 | 宏内核 (Linux) | 微内核 (Minix/QNX) |
|------|--------------|-------------------|
| 子系统通信 | 函数调用（极快） | IPC 消息传递（较慢） |
| 驱动崩溃影响 | 可能 panic 整个系统 | 只影响该驱动进程，可重启 |
| 性能 | 高（函数调用开销） | 较低（IPC 开销大） |
| 复杂度 | 内核内部复杂 | IPC 机制复杂 |
| 稳定性 | 依赖代码质量 | 天然隔离，更健壮 |

### Linux 的折中：模块化宏内核

Linux 在宏内核的基础上引入了**可加载内核模块（LKM）**机制，兼顾了性能和灵活性：

- 核心子系统编译进内核镜像（`vmlinuz`）
- 设备驱动、文件系统支持、网络协议等作为 `.ko` 模块按需加载
- 模块被加载后运行在 Ring 0，与内核共享地址空间（性能无损）
- 模块可以在运行时加载和卸载（不需要重启、不需要重新编译内核）

这使得 Linux 既是宏内核（共享地址空间、函数调用通信），又具备了微内核的部分灵活性（模块化、按需加载）。

> [!conflict] 历史争议：Tanenbaum–Torvalds 论战
> 1992 年，Minix 作者 Andrew Tanenbaum 与 Linus Torvalds 就宏内核 vs 微内核设计展开著名论战。Tanenbaum 认为微内核是未来的方向，宏内核已经过时。30 多年后，Linux 的实践证明：**经过精心工程的宏内核可以既稳定又高性能**，而微内核的 IPC 开销至今仍是实际应用中的瓶颈。然而，微内核的思想在安全攸关系统中（如 seL4 被用于无人机控制、汽车电子）依然活跃。

## 可加载内核模块 (LKM)

### 为什么需要内核模块？

计算机硬件种类繁多，如果所有驱动都编译进内核镜像，内核会膨胀到数百 MB，且每次添加新硬件都需要重新编译内核。

**可加载内核模块**解决了这个问题——它们是可以在运行时动态插入内核的代码片段，无需重启系统或重新编译内核。

### 模块的生命周期

```bash
# 查看当前已加载的模块
lsmod
# 输出示例：
# Module                  Size  Used by
# nvidia_drm             94208  1
# nvidia_modeset       1552384  1 nvidia_drm
# nvidia               56811520  2 nvidia_drm,nvidia_modeset
# btusb                  77824  0

# 加载模块（自动处理依赖）
sudo modprobe bluetooth

# 卸载模块
sudo modprobe -r bluetooth

# 查看模块详细信息
modinfo bluetooth
```

模块存储在 `/lib/modules/$(uname -r)/` 目录下。`modprobe` 会智能解析模块间的依赖关系——加载 `bluetooth` 时，如果它依赖 `btusb`，后者也会被自动加载。

### 模块配置

要在启动时自动加载模块或设置模块参数，在 `/etc/modprobe.d/` 中创建 `.conf` 文件：

```conf
# /etc/modprobe.d/my-modules.conf — 启动时自动加载
# options <module> <param>=<value>  设置模块参数
# blacklist <module>                禁止自动加载
```

### 模块与内核的关系

内核模块运行在 Ring 0，与核心内核共享同一个地址空间。这意味着：
- 一个有 bug 的模块完全有能力破坏内核稳定性（panic）
- 模块可以访问内核内部数据结构和函数（通过 EXPORT_SYMBOL 机制）
- 模块必须与内核版本严格匹配——不同版本的内核 ABI 不兼容

这种"共享地址空间的高风险高收益"正是宏内核设计的缩影。

## 内核内存：不可换出的特权空间

内核的代码和数据有一个重要特性：**绝对不能换出（swap out）到磁盘**。

原因很简单：swap 本身需要内核来执行。如果内核的页面错误处理程序所在的页面被换出了，谁来把它换回来？这将形成一个无法解决的逻辑死锁。

因此 Linux 内核将自己的所有页面标记为**不可换出（unswappable）**。内核内存永远常驻物理 RAM。这意味着：

1. **内核内存占用直接影响系统可用内存**：如果内核泄漏了内存，唯一的解决方法是重启
2. **内核内存映射是直接映射**（或通过 `vmalloc` 使用独立映射区域），不与用户空间的复杂页面回收逻辑混合
3. **`/proc/meminfo` 中的 `Slab`、`KernelStack`、`PageTables` 等字段**反映了内核内存的使用情况

```bash
# 查看内核内存使用
grep -E "^(Slab|KernelStack|PageTables|VmallocUsed):" /proc/meminfo
```

## 跨领域连接

### → Rust：unsafe 是内核的系统调用边界

Rust 的 `unsafe` 块与内核的系统调用共享同一个根本问题：**如何在安全代码和危险操作之间建立可控的边界**？

- `std::fs::read("file.txt")` 最终通过 `libc::read()` 触发 `read()` 系统调用——Rust 的安全封装将裸文件描述符包装为 RAII 类型（`File`），确保文件不会泄漏
- Rust 中的 `unsafe { libc::fork() }` 是对 Linux 内核 `fork()` 系统调用的直接包装——`unsafe` 块标记了"从 Rust 安全世界进入 OS 不安全世界"的边界
- Rust 编译器的安全保证止于用户空间——一旦进入内核（通过系统调用），Rust 无法再提供借用检查或所有权保证。内核自己的代码（C 语言）负责在 Ring 0 内部维护安全
- 内核中的 `copy_from_user()` / `copy_to_user()` 函数验证用户空间指针的有效性——这正是 Rust 借用检查器在编译期做的事情，但内核必须在运行时做（C 没有借用检查器）

```rust
// Rust 安全代码 → std 安全封装 → libc 不安全绑定 → 内核系统调用
use std::fs;

// 这一行代码的执行路径：
// fs::read() → libc::open() → syscall(OPEN) → 内核 open() → 返回 fd
// → libc::read(fd) → syscall(READ) → 内核 read() → 返回数据
// → Drop(fd) → libc::close(fd) → syscall(CLOSE)
let data = fs::read("/etc/hostname").unwrap();
```

参见 [[所有权与借用]]、[[Rust 并发模型]]。

### → JavaScript：运行时封装了系统调用

JavaScript 代码从不直接执行系统调用——它运行在用户空间的运行时（V8 / Node.js / 浏览器）中，由运行时代理所有 OS 交互：

- **Node.js** 的 `fs.readFile()` → libuv → `uv_fs_read()` → 线程池中的 `preadv()` 系统调用。对于网络 I/O，libuv 使用 `epoll`（Linux）/`kqueue`（macOS）/`IOCP`（Windows）包装为统一的事件循环接口
- **浏览器** 的 `fetch()` → 浏览器网络进程 → 内核 socket/bind/connect 系统调用。JS 主线程永远看不到这些底层操作
- **`setTimeout`** 的精度受内核调度器时间片影响——`setTimeout(fn, 0)` 的最小延迟受制于 `CONFIG_HZ` 定义的内核时钟中断频率（通常为 250Hz 或 1000Hz，即 4ms 或 1ms）
- **`SharedArrayBuffer` + `Atomics`** 提供了用户空间的线程同步原语——但它们最终依赖于内核的 futex（fast userspace mutex）系统调用来实现等待/唤醒

参见 [[JavaScript Promise 与异步]]。

### → 软件工程：内核 API 的稳定性承诺

Linux 内核向用户空间暴露了一个**不破承诺**：系统调用接口永不变更。

Linus Torvalds 有一个著名原则：**"Don't break userspace"**——任何对用户空间可见行为的内核变更，只要导致已有程序无法正常运行，就是内核的 bug。这个承诺使得：

- 一个在 Linux 2.6 上编译的二进制程序，可以在 Linux 6.x 上直接运行（二进制兼容性）
- 内核内部可以激进重构（完全公平调度器、io_uring），但外部接口保持稳定
- `/proc` 和 `/sys` 的文件格式也被视为 ABI，受到同样保护

这与软件工程中的**接口稳定性原则**完全一致：实现可以自由变更，但契约不可破坏。区别在于，大多数软件项目的接口稳定性是道德约束，而内核的是**铁律**——违反就会收到 Linus 的公开批评。

> [!note] 稳定接口 vs 稳定实现
> 内核的系统调用表是**稳定接口**（stable API），但内核内部实现**从无稳定承诺**（no stable internal API）。驱动开发者必须接受这一事实：每次内核版本更新，内部 API 都可能发生变化。这是内核能保持长期进化的设计智慧——对外承诺接口稳定，对内允许激进重构。

参见 [[软件工程概述]]。

### → Cocos Creator：引擎运行在用户空间

Cocos Creator 引擎的运行时——无论目标平台是 Web、Android、iOS 还是桌面——完全运行在用户空间。引擎本身只是一个普通的用户进程，所有底层操作最终穿过内核：

- **文件 I/O**：资源加载（纹理、模型、音频）→ `open()`/`read()`/`mmap()` 系统调用
- **网络通信**：WebSocket / HTTP 请求 → `socket()`/`connect()`/`send()`/`recv()` 系统调用
- **GPU 渲染**：OpenGL ES / Vulkan / Metal 的 draw call → GPU 驱动（内核模块）→ 内核 DRM/KMS 子系统 → GPU 硬件
- **音频播放**：AudioContext → ALSA / PulseAudio → 内核音频子系统 → 声卡驱动

当 Cocos 引擎卡顿时，问题可能发生在：用户空间（引擎逻辑 bug）→ 系统调用层（过度 I/O）→ 内核层（驱动效率）→ 硬件层（GPU/CPU 瓶颈）。理解这四层才能有效定位性能瓶颈。

参见 [[Cocos Creator 概述]]、[[图形渲染]]。
