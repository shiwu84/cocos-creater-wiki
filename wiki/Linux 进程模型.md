# Linux 进程模型

> [!abstract] 摘要
> 进程是操作系统中正在执行的程序的实例——它是内核为运行中的代码分配 CPU 时间、内存和 I/O 等资源的基本单元。Linux 进程模型回答一个根本问题：**一块 CPU 如何同时运行数百个程序？** 答案在于三个核心机制：(1) **fork+exec 创建**——每个进程都是复制自身再加载新程序的产物，形成一棵以 PID 1 (init) 为根的进程树；(2) **状态机调度**——进程在 R/S/D/Z/T 五个状态间流转，内核调度器以时间片轮转方式在可运行进程间切换，创造并发的假象；(3) **信号通信**——进程间通过软件中断（SIGTERM、SIGKILL、SIGSTOP 等）进行异步通知和生命周期管理。理解进程模型是理解一切上层抽象（线程、容器、事件循环、异步 I/O）的前提。

## 根本问题：CPU 如何「分身」？

一块物理 CPU 核心在同一时刻只能执行一条指令。但现代系统上同时运行着数百个程序。操作系统通过**进程抽象**解决了这个矛盾：

1. **资源隔离**：每个进程拥有独立的虚拟地址空间——它「以为」自己独占全部内存，实际上内核通过页表将虚拟地址映射到物理内存。
2. **时间分片**：内核调度器以毫秒级时间片在进程间轮转，利用 CPU 的高速切换能力制造并行执行的错觉。
3. **状态保存**：切换时，内核保存当前进程的寄存器、程序计数器、栈指针等上下文，恢复下一个进程的上下文。

> [!note] 关键洞察
> 进程不是「在 CPU 上跑的程序」，而是**内核为程序运行维护的一整套元数据**：内存映射、文件描述符表、信号处理表、调度参数、凭证信息。进程是操作系统管理计算资源的最小抽象单位。

## 进程状态：一台状态机

Linux 进程在以下五种核心状态之间流转：

| 状态 | 符号 | 含义 | 典型场景 |
|------|------|------|---------|
| **运行/可运行** | R | 正在 CPU 上执行，或在运行队列中等待调度 | 活跃计算 |
| **可中断睡眠** | S | 等待事件完成，可被信号唤醒 | 等待终端输入、网络数据 |
| **不可中断睡眠** | D | 等待 I/O 完成，**不可被信号打断** | 磁盘读写、硬件交互 |
| **僵尸** | Z | 已终止，但父进程尚未调用 `wait()` 回收 | 子进程退出后短暂的过渡态 |
| **停止** | T | 被 SIGSTOP 暂停或被调试器挂起 | Ctrl+Z 挂起、gdb 断点 |

### 状态流转

```
fork() → R ⇄ S (等待事件/事件到达)
         R → D (发起 I/O) → R (I/O 完成)
         R → T (收到 SIGSTOP) → R (收到 SIGCONT)
         R → Z (调用 _exit，父进程 wait 后消失)
```

> [!warning] 不可中断睡眠 (D) 的风险
> 进程长时间处于 D 状态通常表示硬件故障或内核驱动 bug。这个状态的设计初衷是保护数据完整性——如果在磁盘写入中途被打断，文件系统可能损坏。但代价是：D 状态进程无法被 kill -9 杀死，只能等待 I/O 超时或重启系统。

> [!tip] 僵尸进程与管理
> 少量僵尸进程是正常的——子进程退出后必然短暂处于 Z 状态，直到父进程 `wait()`。但大量累积的僵尸进程会耗尽进程表（PID 槽位），阻止新进程创建。根本原因是父进程未调用 `wait()` 回收子进程。父进程退出后，init (PID 1) 会收养所有孤儿并自动回收。

## 进程创建：fork + exec

Linux 不提供「从零创建新进程并运行某程序」的单一系统调用。取而代之的是两步模型：

### fork()：克隆自身

```
父进程 (PID=100) ──fork()──→ 子进程 (PID=200, PPID=100)
```

`fork()` 创建一个与父进程几乎完全相同的副本：
- 子进程拥有父进程内存空间、文件描述符、信号处理的完整拷贝
- 子进程从 `fork()` 调用点之后继续执行，但 `fork()` 的返回值不同：父进程得到子进程的 PID，子进程得到 0
- 通过**写时复制 (Copy-on-Write, CoW)**，父子进程共享只读内存页，只有实际写入时才复制页面，极大降低 fork 的内存开销

### execve()：替换自身

```
子进程 ──execve("/bin/ls")──→ ls 进程（PID 不变，程序体替换）
```

`execve()` 用新程序的代码和数据替换当前进程的整个内存空间。PID 不变、文件描述符可保留（取决于 `FD_CLOEXEC` 标志）。这是 fork + exec 组合的精妙之处：fork 保留了环境（文件描述符、环境变量），exec 加载新逻辑。

### 进程树

每个进程都有父进程，这形成一棵以 **init (PID 1)** 为根的树：

```
init (PID 1, 内核启动的第一个用户进程)
 ├── systemd 相关服务
 ├── sshd (PID 500)
 │    └── bash (PID 501, 用户登录 shell)
 │         └── vim (PID 502, 用户执行的程序)
 └── ...
```

`ps l` 可查看 PID（进程 ID）和 PPID（父进程 ID），直观显示这棵树。

## 进程信号：软件中断

信号是 Linux 最核心的进程间通信 (IPC) 机制——它是**异步事件通知**，用软件中断实现。

### 信号的生命周期

1. **生成**：事件发生（Ctrl+C、内核检测到段错误、kill 命令发送）
2. **传递**：内核在下一次调度该进程时投递信号
3. **处理**：进程可以忽略、捕获（执行自定义信号处理函数）、或执行默认操作。SIGKILL (9) 和 SIGSTOP 不能被捕获或忽略

### 核心信号一览

| 信号 | 编号 | 含义 | 可否捕获 |
|------|------|------|---------|
| SIGTERM | 15 | 请求终止（`kill` 默认信号） | 可捕获（优雅退出） |
| SIGKILL | 9 | 强制杀死，立即终止 | **不可捕获/忽略** |
| SIGINT | 2 | 中断（Ctrl+C） | 可捕获 |
| SIGSTOP | 19 | 暂停进程 | **不可捕获/忽略** |
| SIGCONT | 18 | 恢复被暂停的进程 | 可捕获 |
| SIGHUP | 1 | 挂断（终端关闭），常用于守护进程重载配置 | 可捕获 |
| SIGSEGV | 11 | 段错误（非法内存访问） | 可捕获（debug 用） |

### SIGTERM vs SIGKILL

> [!tip] 优雅终止 vs 强制杀死
> `SIGTERM` 是**请求**——进程可以捕获它，执行清理（关闭文件、释放锁、保存状态），然后自行退出。`SIGKILL` 是**命令**——内核立即回收进程资源，不给任何执行清理代码的机会。实践中应先用 SIGTERM，等待超时后再用 SIGKILL。

### 信号掩码

进程可以设置**信号掩码**来阻塞特定信号的投递。被阻塞的信号保持「待处理」状态，直到进程解除阻塞。这是实现临界区保护的一种机制。

## 进程优先级：Niceness

虽然用户不能直接控制 CPU 分配给每个进程的时间，但可以通过 **niceness 值**影响调度器的决策：

- **范围**：-20（最高优先级）到 19（最低优先级），默认 0
- **语义**：「nice」意味着对其它进程友好——nice 值越高，优先级越低，越会把 CPU 让给别人
- **只有 root 可以降低 nice 值**（提高优先级），普通用户只能提高自己的 nice 值（降低优先级）

### 调度器策略

Linux 内核使用**完全公平调度器 (CFS)**，它基于「虚拟运行时间」确保每个可运行进程获得公平的 CPU 份额。nice 值影响每个进程的权重——nice 值为 0 的进程比 nice 值为 5 的进程获得更多 CPU 时间，但系统保证任何进程都不会被饿死。

## /proc 文件系统：进程的自省接口

`/proc` 是 Linux 「万物皆文件」哲学的集中体现——它不是一个真实磁盘文件系统，而是内核在内存中创建的**虚拟文件系统**，暴露内核和进程的内部状态。

```bash
/proc/
├── <PID>/           # 每个进程一个目录，以 PID 命名
│   ├── status       # 进程名、状态、UID/GID、内存、线程数
│   ├── cmdline      # 启动进程的完整命令行
│   ├── environ      # 进程的环境变量
│   ├── fd/          # 打开的文件描述符（符号链接到实际文件）
│   ├── maps         # 内存映射详情
│   ├── stat         # 机器可读的进程统计
│   └── cwd          # 当前工作目录的符号链接
├── cpuinfo          # CPU 信息
├── meminfo          # 系统内存统计
└── uptime           # 系统运行时间和平均负载
```

> [!note] /proc 与监控工具的关系
> `ps`、`top`、`htop` 等工具的数据来源就是 `/proc`。它们是 `/proc` 的友好前端——直接读取 `/proc/<PID>/stat` 可以获得比 `ps` 输出更详细、更结构化的数据。编写监控脚本时，直接读取 `/proc` 往往比解析 `ps` 输出更可靠。

## 进程与线程

**线程**是进程内的执行单元。进程拥有独立的资源（内存空间、文件描述符），而同一进程内的多个线程**共享**这些资源：

| 维度 | 进程 | 线程 |
|------|------|------|
| 内存空间 | 独立，隔离 | 共享进程内存 |
| 创建开销 | 较大（fork+exec） | 较小（只分配栈和寄存器） |
| 通信方式 | IPC（管道、信号、共享内存） | 直接读写共享内存 |
| 隔离性 | 强（一个进程崩溃不影响其他） | 弱（一个线程崩溃可能导致整个进程崩溃） |
| 内核视角 | 独立 task_struct | 也是 task_struct，共享资源指针 |

在 Linux 内核中，线程以**轻量级进程 (Lightweight Process)** 的形式实现——线程和进程使用同一个 `task_struct` 数据结构，区别仅在于资源共享的程度（`clone()` 的 flags 不同）。`ps m` 可以查看每个进程的线程列表。

## 跨领域连接

### → JavaScript：事件循环与进程调度

JS 的**事件循环**运行在单个进程的单个线程上。理解 OS 进程模型后，事件循环的很多特性变得清晰：

- **setTimeout 精度**：`setTimeout(fn, 0)` 并不意味着 `fn` 立即执行——它只是将回调放入宏任务队列。实际执行时间取决于 OS 何时调度该进程的线程、事件循环何时清空微任务队列、以及当前调用栈是否已空。
- **S 状态 = 异步 I/O 的本质**：Node.js 的异步文件读取发生时，发起读请求的线程进入 S 状态（可中断睡眠），等待磁盘 I/O 完成。内核通过 epoll 等机制在 I/O 就绪时唤醒进程——这就是异步 I/O 的底层基础。
- **进程 vs Worker**：浏览器中 `Worker` 创建的是线程而非进程（共享部分资源受限制），Node.js 中 `child_process.fork()` 才是真正的 OS 进程 fork。

参见 [[JavaScript Promise 与异步]]。

### → Rust：从系统调用的类型安全封装

Rust 的 `std::process::Command` 底层调用 fork + exec：

```rust
// 这行代码最终变成 fork() + execve("/usr/bin/git")
Command::new("git").arg("clone").spawn()
```

更关键的是 **Send/Sync trait**——它们不是运行时检查，而是**编译期保证**线程安全：
- `Send`：类型可以安全地转移所有权到另一个线程。对标 OS 层面的跨线程数据传递。
- `Sync`：类型的共享引用可以安全地在多线程间使用。对标 OS 共享内存场景。

Rust 把 Linux 进程模型中「多线程共享内存容易出错」的痛点，在编译期就拦截了。

参见 [[所有权与借用]]。

### → 软件工程：进程抽象的模式启示

进程模型的多个设计对应通用软件工程模式：

- **信号机制 = Observer 模式（OS 级）**：进程注册信号处理器，内核在事件发生时「通知」进程——这是操作系统级别的发布-订阅。
- **进程状态映射服务健康**：R = 正常运行，D = 因 I/O 阻塞（类似 DB 慢查询导致的请求堆积），Z = 资源泄漏（父进程未回收 = 忘记 close/cleanup），T = 暂停（类似熔断器打开状态）。
- **fork + exec 的职责分离**：fork 负责环境准备，exec 负责逻辑替换——类似于工厂方法模式中的创建与使用分离。
- **CoW 的内存优化**：写时复制是性能优化的经典案例——直到真正需要时才付出成本，类似懒加载策略。

参见 [[软件工程概述]]。
